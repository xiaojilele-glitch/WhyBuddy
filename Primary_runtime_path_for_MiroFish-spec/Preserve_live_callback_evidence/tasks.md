根据您提供的模块信息和上下文，以下是针对 **“Preserve live callback evidence”** 模块的任务清单（Tasklist）。该清单旨在指导如何捕获 `MiroFish` 运行过程中的实时回调数据，并确保其作为执行路径的有效证据。

---

### 📋 任务清单：保留实时回调证据 (Preserve live callback evidence)

#### 1. 监控环境初始化
- [ ] **配置回调监听器**：在 `MiroFish` 执行器路径中注入拦截点（Hooks）或监听程序，以捕获实时回调流。
- [ ] **定义证据存储结构**：建立标准化的数据结构（如 JSON 或 Log 格式），确保包含 `event_type`、`timestamp`、`payload`、`status` 和 `correlation_id` 字段。
- [ ] **验证执行器连接**：确认后端执行器（Backend Executor）已就绪，且其产生的活动可被观察流（Observable Stream）捕获。

#### 2. 实时数据捕获与处理
- [ ] **捕获事件元数据**：记录每个回调事件的精确时间戳（或逻辑排序序号）和事件类型（如：Initialization, Execution, Completion）。
- [ ] **Payload 脱敏处理 (Redaction)**：对捕获到的载荷进行脱敏处理，保留摘要信息（Summary），移除或遮蔽敏感数据（Secrets/Tokens）。
- [ ] **关联路径标识**：提取并记录 `correlation_id`，确保捕获的证据与当前 `executor-backed` 任务实例一一对应。
- [ ] **状态跟踪**：实时记录每个回调步骤的返回状态（Success/Failure/Pending）。

#### 3. 证据完整性与持久化
- [ ] **流式写入记录**：随着 `MiroFish` 的运行，将捕获到的回调数据实时写入持久化存储（如 `.evidence` 文件或专用数据库）。
- [ ] **路径轨迹验证**：确保证据中包含足以证明“执行器驱动路径（Executor-backed path）”被实际触发的关键节点日志。
- [ ] **序列完整性检查**：验证事件排序的连续性，防止在高并发或异步流中出现数据丢失或乱序。

#### 4. 交付物验证
- [ ] **生成证据摘要报告**：总结本次执行的回调次数、关键状态转换及最终执行路径证明。
- [ ] **清理临时钩子**：任务完成后，移除或禁用用于监听的拦截点，确保生产环境稳定性。
- [ ] **归档证据文件**：将处理后的脱敏证据关联至父任务上下文，供后续审计或分析。

---

**关键注意点：**
*   **真实性**：证据必须包含执行器特有的指纹信息，以证明路径并非模拟。
*   **安全性**：严格执行载荷脱敏（Redacted payload），严禁将原始密钥或隐私信息写入证据。
*   **可见性**：确保在父模块的“Streaming observable role-agent activity”过程中，这些证据能被实时感知。
