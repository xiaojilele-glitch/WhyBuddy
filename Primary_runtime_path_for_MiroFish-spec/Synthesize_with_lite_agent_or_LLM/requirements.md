# 模块需求文档：Synthesize with lite agent or LLM

## 1. 模块基本信息
- **模块名称**：Synthesize with lite agent or LLM (基于轻量级代理或 LLM 的合成模块)
- **所属项目**：MiroFish 驱动工程 ([GitHub: 666ghj/MiroFish](https://github.com/666ghj/MiroFish))
- **主要路径**：Executor-backed role agent path (执行器驱动角色代理路径) 的备选/回退路径
- **模块定位**：当核心执行路径失效或缺乏实时回调证据时，作为宿主端（Host-side）的兜底方案，生成保守的蓝图（Blueprint）。

## 2. 模块目标与背景
### 2.1 背景
在驱动 `MiroFish` 项目的过程中，首选方案是通过“执行器驱动的角色代理”获取实时执行结果。然而，当执行器环境不可用、无法产生有效产物或缺乏关键的实时回调证据时，系统需要一个降级机制。

### 2.2 目标
1.  **宿主端执行**：将计算负载保留在宿主端，避免对不可用的执行器环境产生依赖。
2.  **生成蓝图**：利用现有的路由元数据（Route Metadata）和仓库引用（Repository References）合成技术蓝图。
3.  **真实性声明**：明确区分“静态合成”与“实时执行”，严禁在缺乏证据的情况下伪造实时执行证明。

## 3. 功能需求

### 3.1 输入数据
模块应接收并处理以下输入：
-   **Route Metadata**：包含预定义的路由逻辑、目标终点及预期的交互流。
-   **Repository References**：对 `MiroFish` 仓库代码结构、API 定义及配置文件的静态引用。
-   **Failure Context**：上游执行器路径失败的原因或缺失的证据类型。

### 3.2 合成逻辑 (Synthesis Logic)
-   **轻量级代理处理 (Lite Agent)**：优先使用本地轻量级代理，根据预设规则对元数据进行解析。
-   **LLM 回退 (LLM Fallback)**：若轻量级代理无法处理复杂逻辑，则调用 LLM 接口，基于上下文生成推理结果。
-   **保守生成策略 (Conservative Blueprint)**：生成的蓝图应基于已知事实。对于不确定的执行环节，应标注为“待验证”或“理论路径”，而非模拟执行结果。

### 3.3 证据管理
-   **证据审计**：模块必须检查生成的产物中是否存在误导性的执行证据（Live Callback Evidence）。
-   **透明度标注**：在输出的蓝图中明确标记“由宿主端合成（Host-side Synthesized）”及“无实时执行证据（No Live Evidence）”。

## 4. 业务流程

1.  **触发阶段**：父模块检测到执行器代理返回无效产物或证据链断裂。
2.  **初始化阶段**：加载 `MiroFish` 仓库的元数据快照。
3.  **合成阶段**：
    -   尝试通过 Lite Agent 生成路径逻辑。
    -   若失败，调用 LLM 进行语义补全和结构化蓝图生成。
4.  **校验阶段**：清理任何可能暗示“实时执行已发生”的伪造字段。
5.  **输出阶段**：返回保守蓝图并标记为“备选路径产物”。

## 5. 技术约束与非功能需求

### 5.1 约束
-   **禁止伪造**：绝对禁止在产物中包含虚假的实时回调（Live Callback）数据。
-   **环境依赖**：模块必须能够在不连接远程 Executor 的情况下独立运行。

### 5.2 性能与可靠性
-   **低延迟**：Lite Agent 的处理延迟应控制在毫秒级；LLM 回退的响应时间应符合系统配置的 Timeout。
-   **确定性**：对于相同的元数据输入，生成的保守蓝图应保持逻辑一致性。

### 5.3 安全性
-   **敏感信息屏蔽**：在参考仓库内容时，应过滤掉任何硬编码的密钥或敏感凭证。

## 6. 验收标准 (Acceptance Criteria)
- [ ] 模块能够在执行器路径失效时被正确触发。
- [ ] 能够成功读取 `MiroFish` 的元数据并生成结构化的蓝图文档。
- [ ] 蓝图中没有任何字段宣称拥有实时执行证据（当该证据确实不存在时）。
- [ ] 产物中明确包含了“Host-side Synthesized”的标识。
- [ ] 在 LLM 介入的情况下，生成的蓝图逻辑与仓库实际代码结构保持一致。
