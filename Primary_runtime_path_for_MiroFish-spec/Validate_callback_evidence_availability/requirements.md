# 模块需求文档：Validate callback evidence availability (验证回调证据可用性)

## 1. 模块概述
### 1.1 模块定义
**Validate callback evidence availability** 是 MiroFish 项目中 `executor-backed role agent` 路径下的核心验证组件。其主要职责是确保执行器（Executor）在运行过程中产生的“回调证据”（Callback Evidence）在关键里程碑节点上是完整、真实且一致的。

### 1.2 背景与目标
在驱动 [MiroFish](https://github.com/666ghj/MiroFish) 通过执行器角色代理路径时，系统不仅关注最终产出物的结构合法性，更依赖实时的回调证据来证实执行过程的可靠性。
本模块的目标是防止“黑盒成功”（即产出物看似正确但无过程凭证的情况），通过强制校验关键节点的证据，决定是完成主流程交付，还是转入保守的回退路径（Fallback path）。

---

## 2. 业务逻辑与功能要求

### 2.1 输入参数
*   **Executor Artifact**: 执行器生成的原始产出物（代码、配置文件或中间数据）。
*   **Live Callback Stream/Logs**: 执行期间捕获的实时回调数据流或证据日志。
*   **Milestone Definition**: 预定义的关键运行时里程碑列表（如：环境初始化、代码注入、动态评估、结果捕获等）。

### 2.2 核心功能描述
1.  **证据存在性校验 (Presence Check)**:
    *   扫描回调记录，确认每一个预设的关键里程碑（Key Milestones）是否都有对应的证据条目。
2.  **内容完整性校验 (Completeness Check)**:
    *   验证每个里程碑对应的回调数据是否包含必要字段（如时间戳、执行状态、负载数据等）。
3.  **数据一致性校验 (Consistency Check)**:
    *   交叉比对执行器产出物的内容与回调证据中的记录。例如：如果产出物宣称完成了步骤 A，回调证据中必须存在步骤 A 的成功记录，且相关参数需匹配。
4.  **状态标记 (Status Tagging)**:
    *   若校验通过：标记执行结果为 `Validated`。
    *   若证据缺失、不完整或存在冲突：标记执行结果为 `Insufficient` (证据不足)。

### 2.3 异常处理与回退机制
*   一旦标记为 `Insufficient`，模块必须立即触发 **Conservative Fallback Path**（保守回退路径）。
*   回退路径应确保系统稳定性，避免使用未经证实的执行结果，可能采取的动作包括：还原配置、调用备用逻辑或抛出安全警告。

---

## 3. 关键运行时里程碑 (Key Runtime Milestones)
*示例（需根据 MiroFish 实际链路调整）：*
*   `MS_01`: Agent 握手与环境上下文加载。
*   `MS_02`: 任务解析与指令分发。
*   `MS_03`: 执行器逻辑实时反馈（Heartbeat/Progress）。
*   `MS_04`: 最终工件序列化完成。

---

## 4. 非功能性要求
*   **低延迟**: 校验逻辑应作为流水线的一环高效运行，不应成为执行路径的性能瓶颈。
*   **可靠日志**: 验证失败时，必须记录详细的缺失点（Missing evidence point），以便开发人员定位执行器故障。
*   **幂等性**: 对同一份产出物和证据集的多次验证应得到一致的结果。

---

## 5. 验收标准 (Acceptance Criteria)
| 编号 | 测试场景 | 预期结果 |
| :--- | :--- | :--- |
| AC-1 | 所有里程碑证据齐全且一致 | 模块输出 `Success`，流程继续。 |
| AC-2 | 缺失任意一个关键里程碑证据 | 模块标记 `Insufficient`，并成功路由至 Fallback 路径。 |
| AC-3 | 回调证据中的参数与产出物冲突 | 模块标记 `Insufficient`，记录冲突详情。 |
| AC-4 | 回调数据格式非法或无法解析 | 视为证据缺失，执行回退策略。 |
