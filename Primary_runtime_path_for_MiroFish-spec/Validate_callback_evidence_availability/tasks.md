# 任务清单：验证回调证据可用性 (Validate callback evidence availability)

## 1. 证据定义与初始化 (Definition & Initialization)
- [ ] **确定关键运行里程碑 (Key Milestones)**：定义 `MiroFish` 在角色代理路径（Role Agent Path）中必须记录的核心阶段（如：初始化、指令解析、逻辑执行、结果生成）。
- [ ] **定义证据标准架构**：明确有效的“回调证据”应包含的元数据字段（如：时间戳 `timestamp`、步骤标识 `step_id`、状态快照 `state_snapshot`、执行器签名 `executor_sig`）。
- [ ] **建立映射关系**：将执行器产出的产物（Artifacts）与预期的回调证据点进行一一映射。

## 2. 存在性与完整性检查 (Existence & Completeness Check)
- [ ] **验证证据存在性**：遍历所有定义的关键里程碑，确认每个点都有对应的实时回调记录。
- [ ] **检查数据完整性**：验证每个回调记录中的必填字段是否非空且格式正确。
- [ ] **识别缺失断点**：如果某个里程碑缺少证据，立即将其标记为“证据缺失（Missing Evidence）”。

## 3. 一致性与逻辑验证 (Consistency & Logic Validation)
- [ ] **时间线序列验证**：确保回调证据的时间戳遵循逻辑执行顺序，不存在时间倒置或冲突。
- [ ] **产物-证据对齐检查**：验证执行器生成的最终产物内容是否与回调路径中记录的中间状态保持逻辑一致。
- [ ] **状态合法性校验**：检查回调中的状态数据是否符合 `MiroFish` 的业务规则约束。

## 4. 结果判定与异常处理 (Decision & Error Handling)
- [ ] **标记执行结果状态**：
    - [ ] 若证据完整一致，标记为 `SUCCESS_WITH_EVIDENCE`。
    - [ ] 若证据缺失、不完整或不一致，标记为 `INSUFFICIENT_EVIDENCE`。
- [ ] **准备保守回退路径 (Fallback Path)**：
    - [ ] 当判定为 `INSUFFICIENT` 时，自动触发保守路径配置。
    - [ ] 提取已有的部分证据（如有），为回退路径提供上下文参考。
- [ ] **记录详细验证日志**：记录具体的验证失败原因（例如：“步骤 B 证据缺失”或“时间戳不匹配”），以便后续调试 `MiroFish` 的执行器路径。

## 5. 输出与交付 (Output & Handoff)
- [ ] **生成验证报告摘要**：为父模块提供一个布尔值结果及对应的状态报告。
- [ ] **持久化存证**：将验证通过的实时回调证据保存至 `MiroFish` 的执行历史仓库中，以备审计。

### 模块目标回顾：
*   **输入**：执行器生成的产物、运行时的实时回调流。
*   **核心逻辑**：通过匹配里程碑与证据，确保执行过程透明且可追溯。
*   **输出**：验证状态（充足/不足）及对应的执行路径决策（主路径/回退路径）。
