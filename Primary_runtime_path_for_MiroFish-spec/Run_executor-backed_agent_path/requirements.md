# 模块需求文档 (PRD) - Run executor-backed agent path

## 1. 模块概述

### 1.1 模块名称
Run executor-backed agent path（运行由执行器支持的代理路径）

### 1.2 模块描述
本模块是运行时流转的核心执行引擎。它负责将 MiroFish 的任务请求通过后端执行器（Backend Executor）转化为具体的角色代理（Role-agent）行为。在执行过程中，模块需实现活动状态的流式传输（Streaming）以及实时回调证据（Live Callback Evidence）的持久化，确保整个执行过程透明、可追溯。

### 1.3 项目背景
*   **目标路线**：通过执行器支持的角色代理路径驱动 [MiroFish](https://github.com/666ghj/MiroFish) 项目，并保留实时回调证据。
*   **父模块上下文**：协调端到端的运行时流程，涵盖从 MiroFish 任务接收、路径路由、执行监控、证据捕获到最终产物验证的全过程。

---

## 2. 功能需求

### 2.1 执行器交互与任务分发
*   **路径激活**：接收来自父模块的路由指令，激活预选的角色代理路径。
*   **后端执行器集成**：调用底层执行器（如容器化环境或任务队列），启动 MiroFish 相关逻辑。
*   **上下文注入**：将任务请求参数、仓库上下文及配置信息准确传递至执行环境。

### 2.2 流式活动观察 (Observable Role-Agent Activity)
*   **实时流输出**：在执行过程中，通过 WebSocket 或 SSE（Server-Sent Events）实时向前端或监控系统推送角色代理的状态变化（如：思考中、调用工具、生成响应）。
*   **颗粒度控制**：记录并展示执行过程中的每一个关键动作（Action）及对应的输入输出。

### 2.3 实时回调与证据保留 (Live Callback Evidence)
*   **拦截与记录**：实时捕获执行过程中的回调信号（Callbacks），包括 API 调用、外部系统交互、文件操作等。
*   **证据持久化**：将捕获的实时回调数据作为“证据链”进行保存，确保在任务完成后可供审计或验证。
*   **状态同步**：确保回调信息与当前的执行步骤严格对应。

### 2.4 异常处理与状态报告
*   **错误捕获**：实时监控执行过程中的崩溃、超时或逻辑错误。
*   **中断处理**：支持任务的主动停止，并妥善保存已产生的中间证据。

---

## 3. 非功能需求

*   **实时性**：活动流传输的延迟应控制在毫秒级，确保用户能实时观察到代理的进度。
*   **可靠性**：证据保留机制必须具备幂等性或强一致性，避免因网络波动导致证据丢失。
*   **可扩展性**：支持不同类型的 Role-agent 路径，能够灵活适配 MiroFish 未来的功能扩展。

---

## 4. 业务逻辑流程

1.  **初始化**：接收父模块传递的 MiroFish 任务包。
2.  **环境准备**：后端执行器准备运行时环境。
3.  **循环执行**：
    *   代理输出动作 -> 捕获并推送到流。
    *   触发生命周期回调 -> 拦截回调并记录证据。
    *   检查任务是否完成。
4.  **收尾**：整合所有证据，将控制权交还给验证模块。

## 5. 数据定义 (示例)

| 字段名 | 类型 | 描述 |
| :--- | :--- | :--- |
| `task_id` | UUID | 任务唯一标识符 |
| `stream_chunk` | Object | 当前流式传输的活动片段 |
| `evidence_log` | Array | 收集到的实时回调证据集合 |
| `status` | Enum | running / completed / failed / interrupted |

---

## 6. 验收标准

1.  **成功执行**：能够通过执行器成功驱动 MiroFish 完成预定任务。
2.  **可见性**：在执行期间，前端能够接收到连续的活动流数据。
3.  **证据完整性**：任务结束后，系统能够提供完整的、带时间戳的回调证据报告。
4.  **异常恢复**：当执行器发生异常时，系统能准确上报错误并保留已知证据。
