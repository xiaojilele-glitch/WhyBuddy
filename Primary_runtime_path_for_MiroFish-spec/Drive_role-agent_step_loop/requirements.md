# 模块需求文档：Drive role-agent step loop

## 1. 模块概述
### 1.1 模块定义
本模块是 MiroFish 执行引擎的核心逻辑层。它负责驱动角色代理（Role-Agent）在后端执行器的支持下，按照“观察-规划-交互-综合-验证”的闭环逻辑进行迭代。该模块不仅要完成既定任务，还必须在执行过程中实时保存和流式传输回调证据（Callback Evidence），以确保最终产出物（Artifact）具备可解释性和可审计性。

### 1.2 业务背景
在 MiroFish 的执行路径中，Agent 需要处理复杂的、多步骤的任务。为了保证系统透明度，用户不仅需要最终结果，还需要了解 Agent 在每一个关键节点上“看到了什么”、“想了什么”以及“做了什么”。

---

## 2. 核心目标
1. **自动化闭环执行**：驱动 Role-Agent 自主完成从任务接收到结果验证的全流程。
2. **实时证据链捕获**：在每一步循环中记录详细的上下文、决策依据和工具调用结果。
3. **可审计性保证**：确保每一个最终产出物都可以追溯到具体的回调证据。
4. **后端联动**：通过 Executor 路径无缝调用底层工具与环境。

---

## 3. 功能需求

### 3.1 步骤循环逻辑 (Iterative Loop)
模块必须严格执行以下五步循环，直至任务完成或触发异常：

1.  **观察 (Observation)**：
    *   从环境、用户输入或上一步输出中提取当前状态信息。
    *   记录观察到的关键原始数据作为证据。
2.  **规划 (Planning)**：
    *   基于当前观察结果，制定或调整后续行动方案。
    *   输出逻辑推理过程，说明为何选择该路径。
3.  **工具交互 (Tool Interaction)**：
    *   通过后端执行器（Executor）调用具体工具（如 API 调用、代码执行等）。
    *   捕获工具的原始输入与输出（Payload & Response）。
4.  **综合 (Synthesis)**：
    *   整合工具执行的结果，更新内部状态。
    *   判断当前进度是否符合预期目标。
5.  **验证 (Validation)**：
    *   对当前阶段性产出进行合规性、准确性检查。
    *   决定是进入下一轮循环，还是结束任务。

### 3.2 回调证据管理 (Callback Evidence & Streaming)
*   **流式传输**：在循环进行时，必须通过实时通道（如 WebSocket 或流式 API）向前端或监控端推送每一步的活动摘要。
*   **证据持久化**：
    *   每一步骤产生的 JSON 对象、日志、截图或数据片段需关联唯一的 `RunID`。
    *   证据需包含时间戳、步骤类型、操作状态（成功/失败）及详细 Payload。
*   **可解释性关联**：最终生成的 Artifact 必须包含一个指向完整证据链的引用（Evidence Manifest）。

---

## 4. 业务流程 (Workflow)

1.  **初始化**：接收来自父模块的执行路径指令，初始化 Executor 上下文。
2.  **循环触发**：启动 Step Loop。
3.  **执行并记录**：
    *   执行 Step $N$。
    *   生成 Callback Data。
    *   Executor 返回执行结果。
    *   将结果与 Evidence 写入存储。
4.  **终止判定**：满足任务达成条件或达到最大迭代次数。
5.  **产出交付**：封装最终结果，并附带完整的证据日志链接。

---

## 5. 非功能性需求
*   **可靠性**：Executor 调用失败时应具备重试机制或优雅的回退处理逻辑。
*   **可追溯性**：任何结果必须有据可查，禁止出现“黑盒”决策。
*   **性能**：证据的采集与流式传输不应显著延迟 Agent 的核心推理速度。
*   **安全性**：在回调证据中，应自动脱敏敏感信息（如 API Key、私密 Token 等）。

---

## 6. 验收标准 (Acceptance Criteria)
*   [ ] Agent 能够成功在 MiroFish 环境下完成跨步骤的复杂任务。
*   [ ] 每一轮循环的输入、思考过程、工具调用和输出均有对应的 Callback 记录。
*   [ ] 系统能实时输出当前步骤的流式状态。
*   [ ] 最终产出的 Artifact 能够通过证据链回溯其生成的每一个逻辑环节。
*   [ ] 所有的代码实现需符合项目在 GitHub (666ghj/MiroFish) 的架构规范。
