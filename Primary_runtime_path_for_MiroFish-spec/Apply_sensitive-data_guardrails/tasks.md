### 📋 任务清单：Apply sensitive-data guardrails

#### 1. 策略定义与标准建立
- [ ] **定义敏感数据分类矩阵**：明确需要脱敏的字段类型，包括但不限于 API Keys、Access Tokens、私钥（Private Credentials）、个人身份信息（PII）以及特定环境的秘密（Secrets）。
- [ ] **确定保留标准**：定义哪些技术细节属于“有用证据”（Useful Technical Evidence），例如 API 的端点名称、响应状态码、请求的时间戳或非敏感的逻辑分支。
- [ ] **选定掩码策略**：针对不同类型的数据制定掩码方式（例如：全掩码 `********`、部分掩码 `sk-abc...123` 或占位符 `<REDACTED_API_KEY>`）。

#### 2. 自动化识别与清洗逻辑开发
- [ ] **开发正则表达式库**：编写并测试用于识别常见密钥格式（如 AWS, GitHub, GCP tokens）的正则规则。
- [ ] **实现上下文感知扫描**：开发能够识别环境变量（如 `.env` 结构）和配置文件中硬编码凭据的扫描逻辑。
- [ ] **编写脱敏处理器 (Scrubber)**：实现核心逻辑，在不破坏 JSON/Markdown 结构的前提下，对 `MiroFish` 产出的原始工件进行字符串替换。

#### 3. 执行器路径与回调证据集成
- [ ] **集成至 Role-Agent 路径**：在 Executor-backed 路径的输出阶段插入脱敏中间件。
- [ ] **处理回调摘要 (Callback Summaries)**：针对实时回调（Live Callback）数据，确保在移除敏感认证信息的同时，完整保留请求序列和执行流证据。
- [ ] **验证结构化一致性**：确保脱敏后的数据仍符合 Blueprint Pipeline 预期的结构化格式。

#### 4. 质量保证与安全性校验
- [ ] **执行漏检测试 (False Negative Check)**：使用包含伪造密钥的样本数据运行 pipeline，确认所有敏感信息均被准确拦截。
- [ ] **执行过度脱敏检查 (False Positive Check)**：确保脱敏逻辑没有误删正常的技术参数（如 ID、版本号），以免导致后续分析失效。
- [ ] **对比验证**：对比脱敏前后的工件，确认“技术证据”的完整性能否支撑对 `MiroFish` 执行路径的溯源。

#### 5. 文档与维护
- [ ] **更新脱敏规则文档**：记录当前已涵盖的敏感数据类型和忽略的白名单。
- [ ] **生成脱敏报告**：在工件元数据中记录“已应用脱敏防护栏”的状态，标记处理时间及版本。
