# 需求文档：验证主路径覆盖 (Validate primary route coverage)

## 1. 模块概述
本模块的核心目标是验证 `MiroFish` 项目在“执行器支持的角色代理路径（Executor-backed role agent path）”下的完整性。验证过程必须确保产出物（Artifact）不仅仅是通用的仓库分析结果，而是包含了从执行启动到回调证据保留、产出物生成及多阶段验证的完整闭环。

## 2. 目标路径
*   **目标项目**: [MiroFish (GitHub)](https://github.com/666ghj/MiroFish)
*   **核心路径**: 执行器支持的角色代理路径 (Executor-backed role agent path)

## 3. 详细需求

### 3.1 路径覆盖要求
验证过程必须显式覆盖以下阶段，严禁仅进行通用的代码仓库扫描：
1.  **执行器激活**: 确认由执行器驱动的角色代理已正确初始化并进入预定路径。
2.  **实时回调证据保留**: 在执行过程中，必须捕获并持久化存储实时的回调（Live Callback）数据，作为路径触发的直接证据。
3.  **产出物生成**: 确保在执行路径结束后，系统生成了相应的产出物文件。
4.  **分阶段验证**: 实施从执行开始到结束的阶段性检查，确保每个节点的数据流转符合预期。

### 3.2 产出物验证标准 (Success Criteria)
产出物（Artifact）必须满足以下条件方可判定为“通过”：
*   **结构有效性**: 符合预定义的架构标准，无格式错误。
*   **路径对齐**: 产出物内容必须直接反映执行器路径的运行结果，而非静态扫描结果。
*   **非空检查**: 产出物及关键日志文件不得为空。
*   **证据支撑**: 必须关联到有效的实时回调证据，证明该产出物是由特定路径生成的。

### 3.3 失败判定与回退逻辑
*   如果产出物结构损坏、路径未对齐或缺乏回调证据，则判定为**主路径失败**。
*   根据父模块逻辑，主路径验证失败将触发回退（Fallback）机制或直接报告异常。

## 4. 业务逻辑流程
1.  **启动**: 调用执行器加载 `MiroFish` 代理路径。
2.  **监控**: 监听并记录执行过程中的回调信号。
3.  **提取**: 路径完成后提取生成的 Artifact。
4.  **校验**:
    *   核对 Artifact 是否包含执行轨迹。
    *   匹配回调证据的时间戳与内容。
5.  **定论**: 输出验证结果（成功/失败）。

## 5. 注意事项
*   **避免通用化**: 必须杜绝“仅描述通用仓库结构”的行为。验证重点在于“执行路径”的动态产出。
*   **数据持久化**: 回调证据应具备可追溯性，以便在后续审计中使用。
*   **环境一致性**: 确保执行器环境与 `MiroFish` 项目的运行需求高度匹配。
