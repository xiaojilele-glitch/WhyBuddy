# 任务自动驾驶平台总叙事（2026-04-23）

## 文档目的

本文用于为 WhyBuddy 提炼一版可直接复用的中文总叙事，供以下场景统一口径：

- README 与首页介绍
- 路演、演示与对外表达
- 对内汇报、产品定义与研发对齐

本文不试图重写现有技术事实，而是把当前已经存在的主仓能力，收束为一套更稳定、更容易传播的“任务自动驾驶平台”叙事。

---

## 一、结论先行

WhyBuddy 不应该再被描述为一个“聊天式 AI 演示项目”，也不只是一个“多 Agent 工作流工具”。

更准确的定义是：

**它是一个面向 AI 任务交付的任务自动驾驶平台。用户输入的不再只是 prompt，而是目的地、约束与交付要求；系统输出的不再只是一段答案，而是一条可规划、可执行、可接管、可回放、可审计的任务闭环。**

这一定义有三个关键含义：

1. 产品主语从“对话”升级为“任务送达”。
2. 平台主能力从“回答问题”升级为“规划路线、组织执行、管理风险、交付结果”。
3. 用户体验主界面从“聊天窗口”升级为“驾驶舱”。

---

## 二、产品定位

## 1. 平台定位

WhyBuddy 的目标不是做一个更花哨的 chat UI，也不是把一组 Agent 能力堆成一个控制台，而是构建一个：

**runtime-first、mission-first、evidence-backed 的任务自动驾驶平台。**

其中三层含义分别是：

- `runtime-first`：重点不是“如何画流程图”，而是“任务是否真的被执行、是否可恢复、是否可观测、是否可信”。
- `mission-first`：对用户暴露的核心对象不是 node、prompt 或 agent，而是任务、路线、进度、接管点与交付结果。
- `evidence-backed`：任何自动执行都不是黑箱；平台必须保留日志、产物、审计、血缘、回放与原因解释。

## 2. 一句话定义

**任务自动驾驶平台，是把用户目标转成可规划路线、可组织车队、可在真实环境执行、可在关键节点请求接管、并可全程回放审计的 AI 任务系统。**

## 3. 它解决的核心问题

传统 AI 产品大多停留在“会回答”，但企业与团队真正关心的是：

- 任务能否被稳定送达
- 过程中是否可见、可控、可接管
- 是否能连接真实工具、真实执行环境与真实交付物
- 出问题后是否能复盘、审计、追责与优化

任务自动驾驶平台回答的不是“模型会不会说”，而是：

**这个任务能不能以可控风险到达目标。**

## 4. 与“任务操作系统”口径的关系

现有 README 中的 “task operating system” 口径仍然成立，但它更偏底座描述。

可以这样理解两者关系：

- `任务操作系统`：强调平台具备任务对象、运行时、执行器、回放、审计等基础设施。
- `任务自动驾驶平台`：强调用户价值与产品体验，即平台会帮用户规划路线、组织执行，并在边界处请求接管。

一句话说：

**任务操作系统描述的是底层能力，任务自动驾驶平台描述的是上层产品承诺。**

## 三、核心对象

为避免继续混用 chat、workflow、mission、agent 等多套叙事，建议对外统一为以下核心对象。

| 核心对象 | 产品含义 | 用户看到的是什么 | 当前主仓承接 |
| --- | --- | --- | --- |
| `Destination` 目的地 | 用户真正想达成的目标状态，包含目标、约束、成功标准、交付要求 | “我要去哪、交付什么、受什么限制” | 当前主要由任务发起输入、`/api/tasks`、mission 创建参数承接 |
| `Route` 路线 | 平台为达成目的地生成的执行方案，包含阶段、候选路径、风险点、时间与成本预估 | “为什么这样走、还有哪些备选” | 当前由 `ExecutionPlan`、workflow runtime definition、任务阶段与 operator 动作局部承接 |
| `Fleet` 车队 | 为这条路线临时组织的角色、Agent、工具、技能与外部执行单元 | “谁在协同完成这件事” | 当前由动态组织、skills、MCP、A2A、guest agents、Web-AIGC 节点族共同承接 |
| `Drive State` 驾驶态 | 任务当前处于理解、规划、执行、审核、阻塞、接管、重规划还是已送达 | “现在进行到哪一步了” | 当前由 mission 状态、workflow runtime 状态、operator state、executor 状态共同承接 |
| `Takeover Point` 接管点 | 需要人类澄清、确认、审批、补充上下文或中止执行的边界点 | “什么时候需要我出手” | 当前由 HITL、`selection`、`confirm_judge`、`param_collection`、任务决策接口与 operator actions 承接 |
| `Evidence Pack` 证据包 | 支撑任务可信交付的日志、产物、状态、审计、血缘、成本、回调记录 | “为什么我能相信这个结果” | 当前由 artifacts、telemetry、audit、lineage、executor callback、runtime events 承接 |
| `Replay` 回放 | 对一次任务执行全过程的时间线、快照与证据复盘能力 | “这件事到底是怎么完成的” | 当前由 `/api/replay`、ReplayPage、snapshot/export/verify 链路承接 |

建议统一理解：

- 对用户层，主对象应是 `Destination / Route / Fleet / Drive State / Takeover / Evidence / Replay`。
- 对系统层，现阶段仍可继续沿用 `mission / task / workflow instance / execution plan` 作为内部实现对象。
- 也就是说，**产品叙事升级，不等于立刻推翻当前主仓数据模型。**

## 四、L1-L5 自动驾驶分级边界

L1-L5 的作用不是营销，而是明确平台承诺边界，防止把“自动化辅助”误讲成“全自动黑箱”。

| 等级 | 定义 | 系统负责什么 | 人负责什么 | 典型边界 |
| --- | --- | --- | --- | --- |
| `L1` 辅助驾驶 | 系统理解目标、拆解任务、给出建议路线 | 提供结构化建议、候选路线、推荐角色与下一步动作 | 人决定并主导执行 | 更像“高级任务副驾” |
| `L2` 协同驾驶 | 系统可自动执行标准步骤，但关键节点需要人确认 | 自动跑通部分标准链路，遇到风险、缺信息、越权动作时请求接管 | 人审批关键动作、补上下文、决定高风险操作 | 更像“能开一段，但手不能离方向盘太久” |
| `L3` 有条件自动驾驶 | 在相对稳定的任务域中，系统可完成大部分路线，仅在异常或边界情况下接管 | 自动规划、自动执行、自动重试、自动重规划，并在需要时主动升级给人 | 人处理少量异常、审批和策略边界 | 平台开始具备真正的任务送达能力 |
| `L4` 限定域自动驾驶 | 在明确边界、明确工具集、明确数据与权限策略下，系统可端到端完成任务 | 在限定场景中闭环执行，默认不需要人工持续盯守 | 人只做策略设定、抽检与例外处置 | 适合标准化、受控、高频业务域 |
| `L5` 开放域全自动驾驶 | 面向开放世界、开放任务、开放风险的全面自治 | 几乎所有任务均可自主完成且可靠 | 人只定义长期目标 | 当前不应作为现实承诺口径 |

## 当前建议口径

基于主仓现状，建议统一对内对外使用以下判断：

- **当前主仓已经具备明确的 `L2` 能力，并且在部分受控链路上开始具备 `L3` 底座特征。**
- **截至 2026-04-26，Task Autopilot Phase 1 已形成 `18 specs / 54 markdown / 345 top-level tasks / 602 raw tasks` 的闭环基线，并落下 shared / server / client 的第一条 compatibility-first 纵切。**
- 不建议当前就对外宣称平台已经实现全面 `L4` 或 `L5`。
- 可以描述为：**“平台已从 AI 聊天工具跨入任务协同驾驶，并正在向有条件自动驾驶演进。”**

## 当前主仓为什么是 L2 起步、向 L3 演进

因为现在已经真实存在以下能力：

- 自然语言发起任务，而不是只发起聊天
- 任务执行中的等待、恢复、重试、升级与中止
- 人类决策点、审批点与高风险动作边界
- 真实执行器、真实回调、真实产物
- 回放、审计、血缘与权限治理

但同时也要承认：

- `Destination / Route / Drive State / Fleet / Takeover / Evidence` 已经通过 specs 与第一条代码纵切开始显性化，但还需要继续深化为真实 runtime 行为，而不是只停留在 projection/view model。
- L1-L5 能力等级已经形成 specs 口径，但还需要进入 live mission facts、route policy 与 cockpit explanation，成为用户可感知的产品承诺。
- “自动重规划、路线对比、信心与风险评分”已经有合同锚点和部分 UI 消费面，但还需要继续收口到同一个驾驶舱运行体验中。

所以，更准确的判断是：

**底座已足够支撑 L2-L3 叙事，产品层已经完成第一阶段显性化；下一步是把显性化对象继续压实为 runtime、治理、证据与指标闭环。**

## 五、三栏驾驶舱

任务自动驾驶平台的核心界面不应是单栏聊天流，而应是三栏驾驶舱。

这三栏不是简单排版，而是三类核心问题的稳定分工：

- 左栏回答：`我要去哪，为什么这样走`
- 中栏回答：`现在走到哪了，系统正在做什么`
- 右栏回答：`什么时候需要我接管，我为什么能相信它`

### 建议信息架构

| 驾驶舱栏位 | 主要回答的问题 | 建议承载内容 | 当前主仓可映射能力 |
| --- | --- | --- | --- |
| 左栏：`目的地与路线` | 任务目标是什么，为什么选这条路线 | 目标摘要、成功标准、交付物、约束、推荐路线、备选路线、风险提示、车队组成 | `UnifiedLaunchComposer`、任务摘要、任务列表、mission 输入、ExecutionPlan 信息 |
| 中栏：`实时驾驶态` | 系统现在在做什么，是否偏航，进度到哪 | 当前阶段、执行流、Agent/工具状态、阶段进度、阻塞点、中间产物、实时日志、状态时间线 | `OfficeTaskCockpit`、`/tasks`、`TasksCockpitDetail`、任务事件、executor 状态、runtime state |
| 右栏：`接管与证据` | 什么时候要人做决定，为什么结果可信 | 澄清请求、审批请求、风险动作、成本、审计、回放入口、证据摘要、血缘、操作历史 | operator actions、`/api/tasks/:id/decision`、`/api/audit`、`/api/replay`、权限与 lineage 能力 |

### 三栏驾驶舱的设计原则

1. 左栏不要沦为“任务标题列表”，而要承载目的地和路线语义。
2. 中栏不要沦为“日志滚动窗口”，而要成为真正的实时驾驶态。
3. 右栏不要只是“附加面板”，而要成为接管、审计与证据的统一入口。

### 与传统任务页的本质差异

传统任务页通常是“列表 + 详情 + 操作按钮”。

三栏驾驶舱则是：

- 左边负责理解目标与路线
- 中间负责盯执行态
- 右边负责接管与信任

也就是说，它不是“任务详情页排成三栏”，而是**把任务执行、人的角色与系统可信性同时产品化**。

## 六、与传统 chat / agent / workflow builder 的差异

任务自动驾驶平台不是对 chat、agent、workflow builder 的简单替代，而是把它们纳入一个更高层的任务交付框架中。

| 维度 | 传统 Chat | 传统 Agent 产品 | 传统 Workflow Builder | 任务自动驾驶平台 |
| --- | --- | --- | --- | --- |
| 用户输入单位 | prompt / 对话 | 指令 / 委派 | 图、节点、连线 | 目的地、约束、交付要求 |
| 用户预期输出 | 一段答案 | 一个 Agent 帮忙做事 | 一张流程图或自动化脚本 | 一个可送达、可接管、可审计的任务闭环 |
| 系统默认视角 | 对话视角 | Agent 视角 | 编排设计视角 | 任务送达视角 |
| 执行关注点 | 回答质量 | Agent 是否工作 | 图是否可编排 | 路线是否合理、执行是否可控、结果是否可信 |
| 过程可见性 | 低 | 中 | 设计期高、运行期未必高 | 运行期高，强调实时态、证据与回放 |
| 人的角色 | 反复追问 | 监督 Agent | 设计者 / 搭建者 | 驾驶员与接管者 |
| 风险治理 | 弱 | 中 | 通常偏配置治理 | 强，强调审批、权限、审计、异常、回放 |
| 成功标准 | 回答像不像 | Agent 有没有产出 | 流程能不能跑 | 任务是否稳定送达、可复盘、可放大 |

## 更准确的关系描述

建议统一使用以下解释，而不是把它们对立起来：

- `Chat` 是任务自动驾驶中的一种交互方式，不再是产品本体。
- `Agent` 是任务自动驾驶中的一种执行单元，不再是唯一叙事中心。
- `Workflow Builder` 是任务自动驾驶中的一种路线设计或控制面，不再等于整个平台。

因此，任务自动驾驶平台的差异不是“不要 chat / 不要 agent / 不要 workflow”，而是：

**把 chat、agent、workflow 都收编到“任务送达”这条主线之下。**

## 七、与当前 whybuddy 主仓能力的映射

任务自动驾驶叙事不是从零发明，而是对当前主仓能力的一次重新收束。

## 1. 已有能力与叙事映射

| 叙事层 | 当前主仓能力 | 映射结论 |
| --- | --- | --- |
| 统一发起入口 | `UnifiedLaunchComposer`、`TasksCommandDock`、`POST /api/tasks` | 已具备“输入目标并生成任务”的入口形态 |

| 任务运行时 | `MissionOrchestrator`、`ExecutionPlanBuilder`、任务路由、任务投影 | 已具备以 mission 为核心的任务执行主干 |
| 图运行时 | `workflow-runtime-engine`、`/api/workflows/:id/runtime/run|resume|retry|terminate|escalate` | 已具备“路线运行时”的底层控制面 |
| 人类接管 | `selection`、`confirm_judge`、`param_collection`、`/api/tasks/:id/decision`、operator actions | 已具备接管点与等待恢复能力 |

| 证据与回放 | `/api/replay`、ReplayPage、snapshots、export、verify | 已具备任务全过程回放与导出能力 |
| 审计与治理 | `/api/audit`、`auditChain`、权限治理、异常检测、relation links | 已具备可信执行与治理底座 |
| 车队与工具扩展 | `/api/mcp`、`/api/skills`、`/api/a2a`、auto-agent、guest agents | 已具备把外部工具与 Agent 纳入“车队”的底座 |
| Web-AIGC 节点能力层 | built-in + extra adapters、58 份 specs 封板基线 | 已具备丰富路线能力库，不必从零造节点世界 |

## 2. 这套映射说明了什么

它说明当前主仓并不缺“任务自动驾驶平台”的底座，真正欠缺的是：

- 一套统一的产品语言
- 一套明确的 L1-L5 边界
- 一套显性的 `Destination / Route / Fleet / Takeover / Evidence` 结构
- 一套对用户更直接的三栏驾驶舱信息架构

换句话说：

**现阶段的关键工作不是再证明平台“能做很多事”，而是把这些能力组织成一个更强的总叙事。**

## 3. 仍需显性化补齐的产品层能力

| 待补齐项 | 当前状态 | 意义 |
| --- | --- | --- |
| `Destination` 模型 | 目前更多体现在任务输入与任务标题层 | 需要把目标、约束、交付、成功标准提升为一等对象 |
| `Route` 语义 | 已有执行计划与 workflow runtime，但用户侧路线感知不强 | 需要把推荐路线、备选路线、风险与成本显性化 |
| `Fleet` 语义 | 已有 agent、skills、MCP、A2A、guest agents | 需要把“车队”从技术名词提升为用户可理解的组织对象 |
| 驾驶态状态机 | 已有 mission/runtime/operator/executor 多套状态 | 需要统一成面向用户的 Drive State 语言 |
| 接管面统一 | 已有 decision、resume、retry、escalate、approval | 需要把各种接管动作收敛成统一体验 |
| 证据包统一 | replay、audit、lineage、artifacts 已存在 | 需要把它们从多个面板收口为“为什么可信”的统一叙事 |

## 八、统一表达建议

为了避免不同场合继续各说各话，建议统一遵守以下表达原则。

## 1. 推荐表达

- 推荐说“任务自动驾驶平台”
- 推荐说“把目标送达，而不只是返回答案”
- 推荐说“可规划、可执行、可接管、可回放、可审计”
- 推荐说“驾驶舱、路线、车队、接管点、证据链”
- 推荐说“runtime-first 的任务交付系统”

## 2. 不推荐表达

- 不建议继续把产品主叙事说成“聊天机器人”
- 不建议把产品缩窄成“Agent 市场”或“Agent 编排器”
- 不建议把产品只定义为“流程图工具”或“工作流搭建器”
- 不建议在没有边界说明的情况下宣称“全自动完成一切任务”

## 3. 建议的总口径

最稳妥的统一说法是：

**WhyBuddy 正在从“可观测、可执行的任务操作系统”升级为“可规划路线、可组织车队、可在边界处接管的任务自动驾驶平台”。**

这个说法既继承了当前主仓事实，也为后续产品演进留出了空间。

## 九、可直接复用文案

## 1. README 短版

WhyBuddy 不是一个 chat playground，也不只是一个 Agent 工作流工具。它是一个任务自动驾驶平台：用户输入目标、约束与交付要求，系统自动规划路线、组织执行、在关键节点请求接管，并最终交付可回放、可审计的任务结果。

## 2. 路演开场版

我们做的不是让 AI 多说几句，而是让任务真的到达。WhyBuddy 把用户输入从 prompt 升级为目的地，把系统输出从答案升级为路线、执行、接管与交付。你看到的不再只是一个聊天窗口，而是一个驾驶舱：左边看目标和路线，中间看实时执行，右边看接管、风险和证据。它的核心价值不是“像不像人在思考”，而是“任务能不能在可控风险下被送达，并且事后可复盘、可审计、可优化”。

## 3. 对内汇报版

WhyBuddy 的下一阶段主叙事，不应继续停留在“多 Agent 办公产品”或“任务操作系统”两个层面，而应升级为“任务自动驾驶平台”。现有主仓已经具备任务运行时、图运行时、真实执行器、等待恢复、人类接管、回放、审计、血缘、MCP、skills、A2A 与 Web-AIGC 节点库等核心底座。接下来的重点不是继续堆能力点，而是把这些能力显性化为统一的目的地、路线、车队、驾驶态、接管点与证据链语言，并收口到三栏驾驶舱体验中。

## 十、最终判断

如果要用一句话概括这份文档的核心判断，那就是：

**WhyBuddy 的真正差异，不是“能调用多少模型、多少节点、多少 Agent”，而是它正在把 AI 从一次次对话，推进成一套有路线、有执行、有边界、有证据的任务自动驾驶系统。**
