# 需求文档：任务自动驾驶可解释性与遥测层

## 目标

定义 WhyBuddy 在“任务自动驾驶”中的可解释性与遥测层，使系统不仅能自动规划和执行任务，也能持续回答用户最关心的几个问题：

- 现在在做什么
- 为什么推荐这条路线或这个动作
- 还剩哪些步骤
- 当前有哪些风险
- 系统有多大把握
- 这些判断有什么证据
- 实时运行信号是否健康

本 spec 的重点不是重新建设一套独立监控平台，而是在现有 `mission / workflow / runtime events / replay / audit / observability` 基础上，定义一层面向任务自动驾驶体验的解释与遥测投影。

## 背景

当前仓库已经具备以下相关基础：

- mission-first 的任务发起与任务工作台；
- Route、Drive State、Fleet、Takeover Point、Replan 等自动驾驶产品对象；
- Mission Runtime 与 workflow runtime 的执行主线；
- runtime events 到 replay / audit 的最小桥接；
- review / audit / verify / revise 等质量与治理链路；
- HITL、decision、approval、wait-resume、escalate 等人工接管能力；
- monitoring / replay / audit / lineage 的关联查询与兼容入口。

这些能力已经能够支撑“任务在运行”，但用户仍然需要一层稳定解释：

- 当前状态不应只是底层节点名或日志；
- 推荐路线不应只有结果，还应说明为什么推荐；
- 风险不应只在失败后出现，而应随路线和执行过程持续提示；
- 置信度不应只是一个孤立分数，而应能说明来源与影响；
- 证据不应被埋在日志和审计记录里，而应能被按当前上下文提示；
- 实时信号不应与审计、回放割裂。

因此，本 spec 定义的“可解释性与遥测层”是任务自动驾驶的信任层、状态解释层和证据提示层。

## 范围

本 spec 包含：

- 当前状态解释；
- 推荐原因解释；
- 剩余步骤解释；
- 风险提示；
- 置信度表达；
- 证据提示；
- 实时状态信号；
- 与现有 observability / audit / runtime events 的兼容口径；
- 可解释对象与任务自动驾驶对象的映射关系。

本 spec 不包含：

- 新建一套独立 telemetry backend；
- 替代现有 replay / audit / monitoring / lineage；
- 重写 Mission Runtime 或 workflow engine；
- 定义完整驾驶舱 UI 布局；
- 对所有底层 runtime 事件做一次性全量改造。

## 核心术语

### 解释层

解释层用于把底层任务事实、路线事实、运行时事件、审计证据和实时信号，翻译成用户可理解的状态说明、原因说明、风险说明和下一步说明。

### 遥测层

遥测层用于归一化任务自动驾驶需要消费的实时状态信号，例如状态变化、节点进展、等待输入、风险变化、置信度变化、证据更新、接管请求、重规划触发等。

### 解释卡片

解释卡片是面向用户展示的解释单元，可以用于表达当前状态、推荐原因、风险、置信度、剩余步骤或证据提示。

### 证据提示

证据提示不是完整证据库，而是在当前上下文中把最相关的日志、产物、runtime event、audit entry、replay snapshot 或 lineage 线索提示给用户。

### 实时状态信号

实时状态信号是任务执行过程中可被驾驶舱、任务详情、接管面板、回放和审计消费的状态变化事实。

## 需求

### 需求 1：系统必须提供当前状态解释

系统必须能基于 `Drive State`、Mission Runtime、workflow runtime、runtime events、review / audit 状态，为用户生成当前状态解释。

当前状态解释至少应回答：

- 当前任务处于哪个高层 `Drive State`；
- 系统正在做什么；
- 当前动作属于哪条 Route 或哪个阶段；
- 当前由哪些 Fleet 角色、节点或执行器承担；
- 为什么当前停留在该状态；
- 下一步预计会发生什么。

验收标准：

- 状态解释不得只展示底层节点名或技术日志。
- 状态解释必须能关联到真实 mission / workflow / runtime 信号。
- 当状态来自推断时，必须保留推断来源或触发信号。
- 状态解释必须能在实时视图、回放视图和审计视图中保持语义一致。

### 需求 2：系统必须解释路线推荐与动作推荐原因

系统必须为路线推荐、路线选择、默认动作、接管建议、重规划建议提供推荐原因。

推荐原因至少应覆盖：

- 为什么该路线或动作适合当前目标；
- 与候选路线或候选动作相比的差异；
- 成本、时间、质量、风险、自动化程度的权衡；
- 当前自动驾驶等级对推荐结果的影响；
- 是否依赖用户确认、权限授权或预算确认。

验收标准：

- 推荐原因必须能关联 `Route`、`Destination`、`Risk`、`Confidence` 与任务上下文。
- 不得只给“系统推荐”这类无依据文案。
- 当推荐原因来自策略规则、历史经验或模型推断时，应标明解释来源类型。
- 高风险推荐必须带证据提示或接管提示。

### 需求 3：系统必须提供剩余步骤解释

系统必须向用户解释任务还剩哪些主要步骤，而不是只展示“执行中”。

剩余步骤解释至少应包含：

- 当前所在阶段；
- 已完成阶段；
- 正在执行阶段；
- 待执行阶段；
- 可能被跳过、合并或重规划的阶段；
- 每个剩余步骤的预期产物、风险和是否可能需要接管。

验收标准：

- 剩余步骤必须来源于 Route、workflow 投影或 runtime projection。
- 执行中发生重规划时，剩余步骤必须更新，并说明变化原因。
- 并行步骤必须能被解释为主线步骤与并行支路，而不是混成无序日志。
- 回放时必须能还原当时用户看到的剩余步骤或等价解释。

### 需求 4：系统必须提供风险提示

系统必须在规划、执行、复核、重规划、接管过程中持续提供风险提示。

风险提示至少应覆盖：

- 目标歧义风险；
- 数据缺失或证据不足风险；
- 成本或时长超限风险；
- 权限、合规、安全风险；
- 外部工具或执行器失败风险；
- 结果质量不达标风险；
- 审计与可追踪性不足风险。

验收标准：

- 风险提示必须说明风险类型、严重程度、触发原因、影响范围和建议动作。
- 高风险提示必须能触发 Takeover Point、降级执行或 Replan。
- 风险变化必须能进入 runtime evidence，并可被 replay / audit 消费。
- 风险提示不得只在失败后出现，应支持执行前和执行中的前置提示。

### 需求 5：系统必须提供置信度表达

系统必须提供面向任务自动驾驶的置信度表达，说明系统对当前理解、路线、执行与结果的把握程度。

置信度至少应拆分为：

- 目标理解置信度；
- 路线可行性置信度；
- 执行完成置信度；
- 结果质量置信度；
- 证据充分性置信度。

验收标准：

- 置信度不得只作为单一分数展示，必须能提供阶段性解释。
- 低置信度必须能触发澄清、接管、降级或重规划策略。
- 置信度变化必须有来源，例如缺失信息、执行失败、审计失败、证据补齐或用户确认。
- 置信度表达必须避免伪精确，可以使用等级、区间或标签。

### 需求 6：系统必须提供证据提示

系统必须将当前解释与相关证据关联起来，使用户知道“为什么系统这样说”。

证据提示至少应支持关联：

- runtime event；
- replay timeline；
- audit entry；
- artifact；
- log 摘要；
- decision / approval 记录；
- review / verify 结果；
- lineage 关联线索。

验收标准：

- 证据提示必须能说明证据类型、证据来源、关联对象和更新时间。
- 证据提示可以只展示摘要，但必须能跳转或关联到完整证据入口。
- 不得把“有 evidence 文案”误写成“所有 runtime 事件已经全量进入 lineage”。
- 当证据不足时，系统必须明确提示证据不足，而不是伪造确定性。

### 需求 7：系统必须定义实时状态信号

系统必须定义任务自动驾驶可解释性所需的实时状态信号目录。

实时状态信号至少应覆盖：

- `drive_state.changed`：高层驾驶状态变化；
- `route.recommended`：路线推荐生成；
- `route.selected`：用户或系统选择路线；
- `route.replanned`：发生重规划；
- `step.progressed`：步骤进展变化；
- `risk.changed`：风险变化；
- `confidence.changed`：置信度变化；
- `evidence.updated`：证据更新；
- `takeover.requested`：请求接管；
- `takeover.resolved`：接管完成；
- `runtime.health_changed`：运行健康信号变化。

验收标准：

- 实时状态信号必须能映射到现有 runtime events、projection 或 audit/replay 消费面。
- 不要求当前立即建立新总线，但必须定义信号语义和最小载荷。
- 信号命名应与现有 observability event catalog 保持兼容，不得制造平行体系。
- 对于尚无直接 runtime 事件的信号，必须允许通过 projection 或组合事件推导。

### 需求 8：系统必须定义解释对象的最小载荷

系统必须为解释层定义最小可用数据结构，使前端、服务端投影、回放和审计可以稳定消费。

最小载荷应包括：

- 解释对象唯一标识；
- 解释类型；
- 关联 mission / workflow / route / step / runtime event；
- 标题与摘要；
- 原因说明；
- 置信度；
- 风险等级；
- 证据引用；
- 建议下一步；
- 生成时间与更新时间。

验收标准：

- 解释对象不得只存在于前端临时字符串中。
- 首阶段可以由 view model 生成，但必须保留后续服务端 projection 的扩展口径。
- 同一解释对象应能被驾驶舱、任务详情、接管面板、回放和审计复用。
- 解释对象必须支持失效、更新和被重规划替换。

### 需求 9：系统必须兼容现有 observability / audit / runtime events

系统必须明确可解释性与遥测层是现有观测、审计和运行事件能力的上层投影，而不是替代系统。

兼容要求：

- runtime events 继续作为执行事实来源；
- replay 继续作为时间线与回放承接面；
- audit 继续作为关键治理与人工决策承接面；
- monitoring 继续作为兼容读取与健康投影入口；
- lineage 作为关联线索和后续增强方向，不被夸大为已全量承接。

验收标准：

- 文档不得声称“单一 telemetry 总线已经承接全部事件”。
- 文档不得声称“所有 runtime 事件已统一直写 lineage”。
- 可解释层新增信号必须能说明来自真实 runtime、projection、audit 或 replay。
- 高风险解释必须能进入 audit 或至少关联 audit entry。
- 高风险解释进入 audit 的规则必须至少覆盖 `权限 / 预算 / 合规 / 外部副作用 / 风险接受 / 路线切换 / 重规划` 七类触发器，并区分“直接写入 audit”与“关联现有 audit entry”两种落点。
- 每类高风险解释的 audit 规则必须定义最小字段，至少包括 `explanationId / explanationType / sourceType / mission-workflow refs / route refs / decision refs / risk or threshold action / evidence refs / trigger reason / recordedAt`。

### 需求 10：系统必须支持回放与审计中的解释复原

系统必须确保关键解释能够在任务完成后被回放和审计复原。

至少应支持复原：

- 当时的当前状态解释；
- 当时推荐路线或动作的原因；
- 当时的风险提示；
- 当时的置信度变化；
- 当时的剩余步骤摘要；
- 当时的接管提示与用户选择；
- 当时可见的证据提示。

验收标准：

- 关键状态变化不能只存在于浏览器内存。
- 对用户决策有影响的解释必须保留可审计记录或可重建事件。
- 回放中的解释应与原执行时间线对齐。
- 审计查询必须能关联解释对象与 mission / workflow / runtime event / decision。
- replay 解释复原规则必须定义时间线槽位，至少覆盖 `current_state / recommendation / remaining_steps / risk / confidence / takeover / evidence_update / replan_change`。
- replay 解释复原规则必须定义排序、版本失效与缺失回退口径，明确 `active / superseded / resolved` 的展示规则，以及当缺少原生事件时如何使用 projection snapshot 或组合事件重建并标注来源。

### 需求 11：系统必须支持分阶段落地

可解释性与遥测层必须允许分阶段实现，避免要求一次性重构 runtime。

推荐阶段：

- 第一阶段：文档定义与术语统一；
- 第二阶段：前端 view model 投影当前状态、剩余步骤、风险和证据提示；
- 第三阶段：服务端 projection 生成可复用解释对象；
- 第四阶段：runtime events 扩展关键解释信号；
- 第五阶段：replay / audit 完整复原解释时间线。

验收标准：

- 第一阶段不得要求重写底层 runtime。
- 第二阶段允许部分解释由组合信号推导，但必须标明来源。
- 第三阶段以后应逐步减少仅前端临时推断。
- 每阶段都必须保持与现有 observability / audit / replay 兼容。

## 验收标准

- 存在一份中文需求文档，明确当前状态解释、推荐原因、剩余步骤、风险提示、置信度、证据提示和实时状态信号。
- 存在一份中文设计文档，说明解释层、遥测信号层、证据关联层与现有 runtime / replay / audit 的关系。
- 存在一份全部未完成的任务清单，用于后续渐进实现。
- 文档明确说明本 spec 不新建独立 telemetry backend，不替代现有 observability / audit / runtime events。
- 文档明确保留当前主仓真实边界，不夸大 lineage、telemetry bus 或全链路观测完成度。
