# 需求文档：任务自动驾驶驾驶舱信息架构

## 目标

定义任务自动驾驶三栏驾驶舱的信息架构，使 `WhyBuddy` 能以“目标 -> 路线 -> 执行 -> 接管 -> 证据”的方式承载复杂任务全过程，并与现有 `mission-first`、`Mission Runtime`、`HITL`、`replay`、`audit` 基座保持兼容。

## 需求

### 需求 1：驾驶舱采用三栏主结构

系统应采用稳定的三栏驾驶舱结构，至少包括：

- 左侧：目标与路线
- 中间：执行主视图
- 右侧：接管与证据

该结构应成为任务自动驾驶场景的默认信息架构，而不是附属于聊天页的临时布局。

### 需求 2：左侧聚焦目标与路线

系统应在左侧提供任务目的地、目标摘要、成功标准、约束条件、当前路线、候选路线、阶段进度、风险提示等内容，帮助用户理解“系统准备把任务送到哪里、准备怎么走、当前走到哪里”。

### 需求 3：中间聚焦执行过程

系统应在中间区域展示当前编队、执行阶段、正在运行的节点/角色、中间产物预览、关键上下文、当前状态和重规划信号，使用户能直接看到“系统正在做什么”。

### 需求 4：右侧聚焦接管与证据

系统应在右侧展示用户接管点、待确认事项、澄清问题、预算或权限决策、审计摘要、回放入口、证据索引、成本与风险信息，使用户能在需要时快速介入，而不是被动等待系统失败后再补救。

### 需求 5：与 mission-first 基座兼容

系统应能将驾驶舱信息架构映射到现有 `mission-first` 底座，包括 `mission`、`task`、`workflow`、`projection`、`session` 等对象，而不是建立与现有任务体系割裂的新模型。

### 需求 6：与 Mission Runtime 状态兼容

系统应支持将 `Mission Runtime` 与 `workflow runtime engine` 中的阶段、节点状态、等待输入、恢复执行、分支选择、执行完成、失败和重试等状态转换为驾驶舱可消费的信息块。

### 需求 7：与 HITL 决策链兼容

系统应支持将现有 `HITL`、`selection`、`param_collection`、`user_input`、`confirm_judge` 等等待与决策场景统一纳入“接管点”视图，而不是让用户分别进入多个孤立界面处理人工输入。

### 需求 8：与 replay / audit 证据链兼容

系统应提供与 `replay`、`audit`、`lineage` 等能力的稳定挂接方式，使驾驶舱可以从执行视图直接跳转或展开相关时间轴、审计记录、数据血缘与关键事件证据。

### 需求 9：默认以任务推进为中心，而非聊天回合为中心

系统应将“任务阶段推进、路线推进、交付推进”作为主组织方式，而不是以消息气泡和对话回合堆叠作为主界面骨架。对话内容可以存在，但应作为执行过程的一部分，而不是唯一主视图。

### 需求 10：支持渐进增强

系统应允许先以信息架构层收敛桌面端主界面，再逐步补充移动适配、3D 可视化、批量任务切换、多任务总览等增强能力，但首版信息架构必须先保证单任务驾驶舱闭环成立。
