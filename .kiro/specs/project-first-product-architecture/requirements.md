# 需求文档：Project-First 产品架构

## 目标

定义 `WhyBuddy` 从“任务自动驾驶平台”进一步升级为“项目级 AI 自动驾驶操作系统”的产品架构。该 spec 是后续 Project Domain、Project Cockpit、Spec Center、FSD Route Planner、Execution Center 等 spec 的总纲。

核心结论：

> 不要再加入口，要开始加主线。系统的第一层对象从 `任务` 升级为 `项目`，所有问答、澄清、spec、路线、执行、证据和产物都沉淀回项目。

## 需求

### 需求 1：Project 成为一级对象

系统应将 `Project` 定义为用户进入产品后的第一层工作对象，而不是把 `Mission`、`Workflow`、`Task` 或 `Chat` 作为用户的首要心智。

### 需求 2：首页围绕当前项目组织

系统应将首页从“多个入口并列展示”收敛为“当前项目驾驶舱”，让用户第一眼知道当前项目、当前阶段、下一步建议和继续推进入口。

### 需求 3：统一主线链路

系统应提供稳定的项目推进主线：

`创建/选择项目 -> 项目问答与澄清 -> Spec 演化 -> FSD 路线规划 -> 角色执行 -> Runtime 承载 -> 证据和产物沉淀 -> 下一轮演化`

### 需求 4：任务中心降级为执行明细

系统应将任务中心定位为项目下的执行队列、运行详情、接管动作、失败恢复和证据回放界面，而不是新的主发起入口。

### 需求 5：50+ AIGC 节点内聚到 FSD 角色

系统应明确 `50+ AIGC 节点` 是 FSD 角色的内置能力池，由角色在执行过程中调用，不作为独立入口、独立页面主对象或用户需要理解的外部节点层。

### 需求 6：Workflow / Docker 是运行承载层

系统应保留 Workflow、Docker、Browser、Native Runtime 等底层能力，但产品表达上应将它们作为执行承载层，而不是和项目、路线、任务并列的用户入口。

### 需求 7：项目状态驱动信息浮现

系统应按项目阶段展示能力：无项目时展示创建项目；目标不足时展示澄清；spec 形成后展示路线；执行中展示状态、证据和接管；完成后展示产物与下一轮演化。

### 需求 8：兼容现有 mission-first 基础

系统应兼容现有 `mission / workflow / runtime / task / evidence` 工程对象，通过 `projectId` 和投影层渐进接入，不应一次性推翻已有任务自动驾驶基础。

### 需求 9：后置资源池能力

系统应将 Key Pool、Agent Pool、多用户权限等能力放在基础支撑层和后续阶段，第一阶段不应让这些能力抢占用户主线。

### 需求 10：可解释、可回放、可沉淀

系统应确保每次项目推进都有可追踪的来源、路线、执行过程、用户决策、产物、证据和回流结果，避免把项目自动驾驶做成黑盒问答。
