# 需求文档：任务自动驾驶平台定位

## 目标

定义 `WhyBuddy` 从“mission-first 任务系统”升级为“任务自动驾驶平台”时的产品定位，用统一语言说明平台解决什么问题、不解决什么问题、与相邻产品类型的区别，以及对内对外的表达边界。

## 需求

### 需求 1：对外一句话定义统一

系统应提供统一的对外一句话定义，用于 README、官网、演示稿、产品介绍页和销售沟通，确保不同场景下不再混用“聊天工具”“工作流平台”“Agent 集合”等表述。

### 需求 2：与 Chat Playground 的区别清晰

系统应明确说明任务自动驾驶平台不是以“单轮问答 + 工具补充”为核心的 chat playground，而是以“输入目标、生成路线、组织执行、持续校正、交付结果”为核心的目标驱动型产品。

### 需求 3：与 Workflow Builder 的区别清晰

系统应明确说明任务自动驾驶平台不是以“用户手工搭流程、手工连节点、手工维护 DAG”为核心的 workflow builder，而是由系统自动完成路线建议、节点编队和执行治理，用户主要负责给出目标与关键接管决策。

### 需求 4：与 Agent Platform 的区别清晰

系统应明确说明任务自动驾驶平台不是单纯展示多个 Agent、模型、工具和技能的能力市场，而是围绕“任务送达结果”组织 Agent、工具和治理能力的统一运行平台。

### 需求 5：与现有 mission-first 的关系统一

系统应明确说明“任务自动驾驶平台”不是对现有 `mission-first` 方向的推翻，而是其上位产品抽象。现有 `mission / workflow / runtime / review / audit` 能力继续作为底层运行基础，对外升级为 `destination / route / drive state / takeover` 的产品语言。

### 需求 6：平台边界清晰

系统应明确平台当前覆盖的边界，包括目标理解、路线规划、节点编队、执行编排、过程澄清、接管决策、结果交付、回放审计；同时明确不承诺开放域全自动、无限工具自治、无条件正确结果等超出当前能力边界的内容。

### 需求 7：产品承诺分级可扩展

系统应支持后续按自动化等级持续升级产品承诺，但当前定位文档必须避免直接宣称“全自动完成所有复杂任务”，为后续 `L1-L5` 自动驾驶分级 spec 预留空间。

### 需求 8：定位可映射到现有能力

系统应确保定位描述能够映射到现有仓库中的任务运行时、工作流引擎、人机协同、审计回放、A2A 协作、Web-AIGC 编排等基础能力，避免出现只有宣传口径、没有系统承接的空定义。
