# 设计文档：任务自动驾驶平台定位

## 设计概述

本 spec 负责定义“任务自动驾驶平台”的产品定位，不直接承载具体节点实现、运行时接口或页面细节，而是为后续 `core concepts`、`L1-L5`、`cockpit IA`、`route planner` 等 spec 提供统一叙事基础。

该定位层的目标不是替换现有技术实现命名，而是建立一层稳定的产品语言，把现有 `mission-first` 能力重新组织为用户更容易理解和接受的系统表达。

## 核心设计

### 1. 平台定义

对外统一定义为：

> `WhyBuddy` 是一个面向复杂任务的任务自动驾驶平台：用户输入目标、查看路线，让系统执行安全部分，并在人类判断必需时接管。

对内统一理解为：

- 产品层：`destination / route / drive state / fleet / takeover`
- 运行层：`mission / workflow / runtime / review / audit / replay`

产品层负责用户理解，运行层负责工程承接。

### 2. 与相邻产品类型的差异

#### 2.1 与 Chat Playground 的差异

Chat Playground 的核心对象是“对话回合”，默认路径是“提问 - 回答 - 追问”。

任务自动驾驶平台的核心对象是“目标送达”，默认路径是：

1. 输入目标
2. 识别目的地与约束
3. 生成路线
4. 自动组织执行
5. 在关键点请求接管
6. 输出交付结果

因此本平台不以聊天框轮次为中心，而以任务推进状态为中心。

| 对比维度 | 任务自动驾驶平台 | Chat Playground |
| ---- | ---- | ---- |
| 核心对象 | 任务目标、路线推进与结果送达 | 对话回合、提示词与回复 |
| 默认输入 | 目标、约束、交付要求 | 问题、提示词、追问 |
| 默认路径 | 输入目标 -> 规划路线 -> 组织执行 -> 关键点接管 -> 交付结果 | 提问 -> 回答 -> 追问 -> 工具补充 |
| 用户心智 | 我在发起并推进一项任务 | 我在和模型持续对话 |
| 系统焦点 | 任务状态、路线状态、接管状态、交付状态 | 会话上下文、回答质量、工具返回 |
| 交付形态 | 草案、结果包、证据、回放入口 | 文本回复、工具输出、会话记录 |

#### 2.2 与 Workflow Builder 的差异

Workflow Builder 的核心动作是“搭图”，强调人工配置节点、边、条件和分支。

任务自动驾驶平台的核心动作是“发车”，强调系统根据目标自动给出推荐路线、自动分配能力包、自动治理执行过程。用户可以查看和调整路线，但不要求用户从零搭建流程图。

| 对比维度 | 任务自动驾驶平台 | Workflow Builder |
| ---- | ---- | ---- |
| 核心动作 | 发起任务并送达结果 | 搭建流程图并维护分支 |
| 路线来源 | 系统先推荐，用户再确认或微调 | 用户手工连接节点、条件和边 |
| 用户职责 | 提供目标、约束与关键接管决策 | 配置节点、分支、条件、重试与编排细节 |
| 编排方式 | 以系统规划和运行时治理为主 | 以人工搭图和显式流程控制为主 |
| 治理重心 | 预算、权限、风险动作、接管与审计 | 流程正确性、节点配置完整性 |
| 价值主张 | 降低任务推进成本，提高送达稳定性 | 提高流程可配置性与流程复用性 |

#### 2.3 与 Agent Platform 的差异

Agent Platform 更强调能力市场、Agent 列表、工具列表、模型编排和插件生态。

任务自动驾驶平台将这些能力收敛为完成任务的执行编队，不以“有哪些 Agent”作为主要卖点，而以“是否能稳定送达结果”作为主要价值。

| 对比维度 | 任务自动驾驶平台 | Agent Platform |
| ---- | ---- | ---- |
| 核心叙事 | 任务送达、路线推进、接管与交付 | 能力市场、Agent 列表、工具 / 模型生态 |
| 用户先看到 | 目标、路线、驾驶状态、证据 | Agent 卡片、工具清单、插件配置 |
| 能力组织方式 | 围绕当前任务动态编队 | 围绕能力供给做注册、发现和编排 |
| 主价值判断 | 是否把任务稳定送达结果 | 是否提供足够丰富的 Agent / 工具能力 |
| 治理焦点 | 路线风险、接管时机、证据与审计 | 组件接入、能力管理、权限与插件生命周期 |
| 对外产品中心 | 任务、路线、结果包 | Agent、工具、市场 / 生态 |

### 3. 与 mission-first 的关系

现有 `mission-first` 定义保留，作为底层工程哲学：

- 用户不是执行零散动作，而是在推进一个 mission
- 系统不是返回单个答案，而是在运行一条任务链

“任务自动驾驶平台”是在此基础上的产品升级：

- `mission` 对外表达为 `destination`
- `workflow` 对外表达为 `route`
- `runtime state` 对外表达为 `drive state`
- `human approval / clarification` 对外表达为 `takeover point`

| 现有运行对象 | 对外产品语言 | 在定位中的作用 |
| ---- | ---- | ---- |
| `mission` | `destination` | 把“正在推进什么目标”表达为用户可理解的目的地 |
| `workflow` | `route` | 把“如何推进任务”表达为路线选择与路线状态 |
| `runtime state` | `drive state` | 把“当前运行到哪里、是否降级或重规划”表达为驾驶状态 |
| `human approval / clarification` | `takeover point` | 把“何时必须人工介入”表达为统一接管语义 |

这样可以在不推翻既有实现的情况下，升级用户认知与产品包装。

### 4. 平台边界

当前平台明确覆盖：

- 复杂任务目标接收与目标摘要
- 路线推荐与阶段拆解
- Agent / 节点编队与执行分工
- 过程中的澄清、审批、接管与重规划
- 结果交付、证据展示、回放与审计

当前平台明确不承诺：

- 开放域 L5 全自动执行
- 不受预算、权限、上下文限制的无限自治
- 对所有任务一次成功且无需校验
- 让用户直接管理全部底层节点细节

| 边界维度 | 当前平台覆盖 | 当前不承诺 |
| ---- | ---- | ---- |
| 目标理解 | 接收复杂目标、补全约束、产出任务摘要 | 无上下文也能始终正确理解任何开放目标 |
| 路线规划 | 推荐路线、阶段拆解、能力编队建议 | 所有任务域都自动给出稳定最优路线 |
| 执行编排 | 组织执行、澄清、接管、重规划 | 无视预算、权限和策略边界的无限自治 |
| 结果交付 | 输出草案、结果包、证据与回放入口 | 对所有任务一次完成且无需复核 |
| 用户界面 | 提供任务与接管主视角 | 让用户直接操作全部底层节点和编排细节 |

### 5. 对外表达约束

后续 README、官网、介绍材料应统一遵守以下表达约束：

- 可以说“不是对话式 AI，而是任务自动驾驶”
- 可以说“输入目标，系统规划路线并组织执行”
- 可以说“必要时由用户接管关键决策”
- 不应说“完全替代人工”
- 不应说“自动完成任何复杂任务”
- 不应把底层 50+ 节点直接作为产品主入口

| 可以说 | 不应说 | 约束原因 |
| ---- | ---- | ---- |
| 输入目标，系统规划路线并组织执行 | 系统自动完成任何复杂任务 | 当前仅覆盖有治理边界、可接管、可审计的任务链路 |
| 必要时由用户接管关键决策 | 完全替代人工 | 当前产品承诺本身包含 HITL、审批、澄清与交付确认 |
| 不是对话式 AI，而是任务自动驾驶 | 只是一个聊天框加工具 | 平台核心对象是任务送达，不是会话轮次 |
| 系统推荐路线，用户确认关键节点 | 用户需要自己从零搭流程图 | 产品目标是降低任务编排负担，而不是把 DAG 暴露成主入口 |
| mission-first 是底层工程语义，destination / route 是对外产品语言 | 我们已经放弃 mission-first 并完全重写底层模型 | 当前定位是上位产品抽象，不是推翻既有实现 |

### 6. 可复用定位摘要

供后续 specs、驾驶舱文案和内部方案引用的最小定位摘要如下：

- `WhyBuddy` 是面向复杂任务的任务自动驾驶平台，而不是单轮问答式 chat playground。
- 用户主要负责给出目标、约束和关键接管决策，系统负责路线建议、执行组织、过程治理与结果送达。
- 当前平台的真实承诺建立在 mission-first runtime、HITL、审计回放与治理边界之上，不宣称开放域 `L5` 全自动。

## 设计约束

- 不在本 spec 中定义具体页面布局，页面布局由 `autopilot-cockpit-information-architecture` 负责
- 不在本 spec 中定义具体状态机与接口字段，状态机由 `drive-state-and-replan-state-machine` 负责
- 不在本 spec 中定义自动化等级细则，等级定义由 `task-autopilot-levels-l1-to-l5` 负责
- 不在本 spec 中改变现有底层命名，仅定义产品层映射方式
