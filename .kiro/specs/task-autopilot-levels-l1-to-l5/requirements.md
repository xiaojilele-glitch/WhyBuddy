# 需求文档：任务自动驾驶 L1-L5 分级

## 目标

定义 WhyBuddy 在“任务自动驾驶平台”方向上的 L1-L5 自动化分级标准，使产品、设计、运行时、治理、前端驾驶舱与 Web-AIGC 节点编排能够共享同一套等级语言。

本分级不是抽象宣传口号，而是要直接对接当前仓库已存在的 mission-first 系统能力，包括：

- 办公室主壳与 `/tasks` 任务工作台
- 十阶段工作流引擎与 Mission Runtime
- wait / resume / review / audit / replay / lineage 相关能力
- human-in-the-loop、DecisionPanel、任务审批与人工接管链路
- Web-AIGC built-in / extra adapters、runtime governance 与高风险动作治理

本 spec 的目标不是承诺“已经达到 L5”，而是建立一个可执行、可验证、可演进的自动化等级框架。

## 当前系统边界（2026-04-23）

以下事实应作为本分级的现实基础：

- 当前系统已经不是单轮问答页，而是 mission-first 的任务操作系统。
- 当前系统已经具备任务发起、阶段推进、真实执行、回放、审计、人工介入等基本链路。
- 当前系统已经具备 built-in adapters、Web-AIGC extra adapters、wait / resume 控制面、runtime governance、审计与回放桥接。
- 当前系统仍然没有达到开放域全自动执行；在复杂任务、外部系统调用、高风险动作、预算消耗、权限变更等场景中仍需要明确接管策略。
- 当前系统的真实能力成熟度在不同任务域并不一致，不能简单使用单一“自动化程度”覆盖所有任务。

因此，本分级必须支持：

- 按任务类型区分自动化水平
- 按路线阶段区分自动化水平
- 按风险等级定义接管要求
- 按运行时证据验证自动化承诺

## 需求

### 需求 1：系统必须定义统一的 L1-L5 自动驾驶等级

系统必须为任务自动驾驶定义统一的五级自动化等级，并保证这些等级可被产品文案、设计稿、运行时策略、审计与运营统一引用。

五级分级至少包括：

- `L1`：路线建议级
- `L2`：部分自动执行级
- `L3`：标准任务自动闭环级
- `L4`：限定任务域高自动化级
- `L5`：开放任务域全自动级

验收口径：

- 每一级都必须有清晰的能力定义，而不是只给名称。
- 每一级都必须能说明与当前 mission-first 系统的关系。
- 每一级都必须说明不属于该等级的能力边界。

### 需求 2：每一级必须定义自动化能力范围

系统必须明确每个等级允许系统自动完成哪些动作，不允许系统自动完成哪些动作。

每一级至少要覆盖以下能力维度：

- 目标理解
- 路线生成
- 车队组建
- 节点 / Agent 调度
- 执行推进
- 结果复核
- 自动重规划
- 证据记录

验收口径：

- `L1` 必须明确以“建议和辅助”为主，不允许被描述成自动完成任务。
- `L2` 必须明确允许局部自动执行，但关键节点仍需要人工确认。
- `L3` 必须明确允许标准化任务在限定风险内自动闭环。
- `L4` 必须明确只针对限定任务域，而不是开放域。
- `L5` 必须明确当前仅为远期目标，不得在现状中冒充已具备能力。

### 需求 3：每一级必须定义接管要求

系统必须为每一级定义清晰的人机接管要求，使“自动驾驶”不会退化成不可解释的黑盒执行。

接管要求至少覆盖：

- 是否需要用户在启动前确认路线
- 是否允许执行中自动澄清后继续推进
- 是否允许高风险节点自动执行
- 是否必须在预算、权限、发布、外部副作用前请求接管
- 是否允许结果在无人工复核下直接交付

验收口径：

- 每一级都必须有“何时必须接管”的条件说明。
- 接管要求必须能映射到现有 HITL / 决策面板 / review / resume 链路。
- 接管要求必须能支持审计复盘。

### 需求 4：每一级必须定义适用任务范围

系统必须定义每一级的典型适用任务范围，防止把高等级自动化错误套用到不适合的任务。

任务范围至少包括：

- 信息整理与总结类任务
- 文案、方案、结构化交付类任务
- 检索、问答、轻量研究类任务
- 多步骤内容生产类任务
- 带外部副作用的操作类任务
- 高风险治理类任务

验收口径：

- 每一级都必须给出适合与不适合的任务例子。
- 任务范围必须与现有节点 / route family / runtime adapter 能力保持一致。
- 不能把当前尚未稳定实现的任务类型错误写进高等级可用范围。

### 需求 5：每一级必须定义风险边界

系统必须为每一级定义风险边界，用于说明哪些风险可由系统自行处理，哪些风险必须升级为人工接管或人工审批。

风险边界至少包括：

- 成本风险
- 权限风险
- 数据写入 / 删除风险
- 外部系统副作用风险
- 合规与审计风险
- 结果失真与偏航风险

验收口径：

- 风险边界必须能对接 runtime governance、risk-actions、audit / lineage / replay。
- 必须区分“允许自动重试”与“必须人工接管”的失败类型。
- 必须区分“可自动恢复”与“不可自动恢复”的场景。

### 需求 6：等级定义必须能映射到 mission-first 现有系统

本分级必须明确与现有 mission-first 架构的对接方式，不允许另起一套孤立模型。

至少需要映射到：

- `Mission`
- `Workflow`
- `Mission Runtime`
- 十阶段工作流阶段
- Web-AIGC runtime adapter / node family
- human-in-the-loop 与 review / audit / replay

验收口径：

- 必须说明等级是“产品层分级”，不是要求立即重命名底层所有对象。
- 必须说明等级如何投影到当前任务发起、路线推荐、执行推进、接管、复盘链路。
- 必须说明哪些能力依赖后续 specs 才能真正落地。

### 需求 7：等级必须支持运行时判定与证据留存

系统必须允许在运行时记录“当前任务按哪个自动化等级运行”，并为关键自动化行为保留证据。

至少需要记录：

- 任务启动时宣告等级
- 任务是否在中途降级或升级
- 接管发生位置
- 自动重规划发生位置
- 被阻断的高风险动作
- 最终交付是否经过人工确认

验收口径：

- 这些信息必须能够被 replay / audit / lineage / dashboard 消费。
- 等级变更必须可追踪。
- 接管与降级必须有明确原因。

### 需求 8：等级必须支持阶段性落地，而不是一次性全量实现

系统必须允许按阶段推进自动驾驶能力，不要求所有等级同时完整落地。

建议的落地顺序至少应支持：

- 先定义等级语言与策略边界
- 再接入驾驶舱展示
- 再接入 Mission Runtime 运行时标记与接管策略
- 最后再扩展到更高等级任务域

验收口径：

- `L1-L3` 应作为近期可落地主线。
- `L4` 应作为限定场景目标。
- `L5` 应明确为研究 / 远期概念目标。

### 需求 9：等级必须有统一的元数据与降级轨迹设计

系统必须为自动驾驶等级定义最小元数据合同，使 L1-L5 不只是产品文案标签，而是能映射到任务声明、运行时生效等级、降级原因与交付确认状态的结构化信息。

最小字段至少包括：

- `declaredAutopilotLevel`
- `recommendedAutopilotLevel`
- `effectiveAutopilotLevel`
- `levelDecisionSource`
- `levelChangedAt`
- `levelChangeReason`
- `levelChangeDecisionId`
- `levelChangeStageKey`
- `levelTakeoverType`
- `deliveryConfirmedByHuman`

验收口径：

- 必须明确这些字段在 `Mission`、`Workflow / runtime context`、`MissionAutopilotSummary / projection` 中的建议挂载位置。
- 必须说明哪些字段当前可先由 `route / driveState / takeover / recovery / explanation / evidence` 近似表达，哪些字段仍属于后续真实落码目标。
- 必须区分“任务声明等级”“系统推荐等级”和“运行时实际等级”。

### 需求 10：等级必须提供阶段映射、节点族映射与强制接管清单

系统必须提供 L1-L5 与当前 mission-first 主线之间的结构化映射，不允许只停留在概念描述。

至少需要覆盖：

- 十阶段工作流阶段到 L1-L4 自动化动作范围的映射表
- 当前 `Route` 模式到等级语义的映射表
- Web-AIGC 节点族 / route family 到等级边界的映射表
- 首批“必须接管”高风险动作清单

验收口径：

- 必须完整覆盖 `direction / planning / execution / review / meta_audit / revision / verify / summary / feedback / evolution` 十阶段。
- 必须至少覆盖导航节点、执行节点、治理节点三类节点族。
- 必须说明当前仓库在没有独立 `routeFamily` 字段时，如何使用现有 `route.mode` 与治理信号作为过渡锚点。
- 必须明确哪些动作在任何等级下都不能绕过人工确认。

### 需求 11：等级必须提供首批试点清单与统一验收矩阵

系统必须给出分级落地的首批试点清单和统一验收矩阵，避免等级定义无法转化为真实推进顺序。

至少需要包括：

- 哪些任务当前必须停留在 `L1`
- 哪些任务适合优先按 `L2` 推进
- 哪些任务可作为受限 `L3` 首批试点
- 按“任务类型 × 风险等级 × 自动驾驶等级 × 接管要求”的验收矩阵

验收口径：

- 试点清单必须与当前 README / steering / runtime coverage 中已经存在的任务族谱保持一致。
- 试点清单必须明确“可试点”和“仍不可默认开放”的边界。
- 验收矩阵必须能直接服务于后续驾驶舱展示、runtime metadata 与治理策略接线。

## 分级要求明细

### L1：路线建议级

定义：

- 用户输入目标后，系统能够帮助理解目标、补齐结构化约束、推荐可选路线与执行建议。
- 系统可以生成任务草案、路线草案、阶段拆解建议，但默认不直接自动执行高成本或高副作用链路。

自动化能力：

- 自动解析用户目标
- 自动生成 1 到 3 条推荐路线
- 自动标记缺失信息和建议澄清项
- 自动给出推荐节点 / 角色编组

接管要求：

- 启动正式执行前必须由用户确认
- 所有外部副作用动作必须人工确认
- 成本敏感与权限敏感动作必须人工确认

适用任务范围：

- 方案规划
- 任务拆解
- 信息整理
- 研究路线建议
- 内容生产前置规划

风险边界：

- 允许推荐，不允许默认执行高风险动作
- 即使有自动补全，也不得绕过人工最终启动确认

### L2：部分自动执行级

定义：

- 系统可以在用户授权的主路线下自动执行部分标准步骤，但在关键节点、风险节点或不确定节点上必须请求接管。

自动化能力：

- 自动执行低风险检索、总结、结构化生成步骤
- 自动组织常见角色编队
- 自动在低风险节点间推进 workflow
- 在局部失败时自动重试或改写低风险步骤

接管要求：

- 路线启动前通常需要确认
- 涉及预算跃迁、权限升级、外部写操作、宿主动作时必须接管
- 结果交付前建议人工确认

适用任务范围：

- 标准问答与研究
- 报告初稿生成
- 常规方案整理
- 内部资料汇总
- 轻量内容编排任务

风险边界：

- 可自动处理低风险失败与低风险澄清
- 不允许越过权限 / 预算 / 外部副作用边界
- 不允许对高风险节点做默认无人值守执行

### L3：标准任务自动闭环级

定义：

- 对于结构相对稳定、风险可控、证据链完整的标准任务，系统可在限定治理边界内自动完成大部分从路线到交付的闭环，并仅在异常或策略命中时请求接管。

自动化能力：

- 自动解析目标并选择默认路线
- 自动组建角色车队并执行主路线
- 自动进行低到中风险重试、重规划与质量复核
- 自动沉淀执行证据、审计事件与回放轨迹

接管要求：

- 默认仅在预算超阈值、置信度过低、权限不足、外部副作用超界、结果复核失败时接管
- 对标准任务允许在满足策略的前提下直接交付草案或建议结果

适用任务范围：

- 标准化内容生产链路
- 模板化研究与分析任务
- 可复用结构的运营与知识任务
- 有明确定义输入输出格式的 Office / Web-AIGC 任务

风险边界：

- 仅适用于中低风险、可审计、可回放的任务域
- 必须保留完整 evidence / audit / replay
- 出现高风险动作时必须降级为 L2 或触发接管

### L4：限定任务域高自动化级

定义：

- 在明确限定、强治理、强约束的任务域内，系统可以实现高比例无人值守执行，并具备较强的自动恢复、自动重规划和自动复核能力。

自动化能力：

- 在特定场景下自动从目标到交付闭环
- 自动执行多阶段路线选择与重规划
- 自动调用限定能力包、限定节点族与限定外部接口
- 自动根据审计、质量和风险策略触发降级或人工升级

接管要求：

- 日常流程可少接管
- 只在越过场景白名单、权限白名单、预算白名单、合规白名单时强制接管
- 必须支持运营强制中断与回放审计

适用任务范围：

- 限定模板化业务流程
- 限定组织内知识任务
- 限定内容生产流水线
- 限定审批链已定义的任务域

风险边界：

- 只适用于白名单场景，不适用于开放任务域
- 需要强策略、强审计、强权限隔离
- 一旦脱离白名单，必须降级到 L3 / L2

### L5：开放任务域全自动级

定义：

- 系统能够在开放任务域中自主完成从目标理解、路线规划、执行、重规划、恢复到交付的完整链路，并在绝大多数情况下无需人工接管。

自动化能力：

- 理论上覆盖开放域复杂任务
- 理论上具备通用策略理解、动态风险控制与高可靠恢复

接管要求：

- 理论上极少接管
- 现实中仍需要保留紧急停止与监管能力

适用任务范围：

- 开放任务域复杂任务

风险边界：

- 当前仓库不应把任何现有能力表述为 L5 已落地
- L5 仅用于远期研究目标、概念框架与未来平台规划

## 与当前系统的验收映射

为避免写成空话，本 spec 在当前仓库中的最低映射口径如下：

- `L1` 对应现有 NL 指挥入口、任务发起、路线建议、人工启动执行
- `L2` 对应现有 mission-first 执行主线中“低风险自动执行 + 关键点接管”
- `L3` 对应现有标准化 Web-AIGC / Office / 内容任务在 runtime governance 下的自动闭环目标
- `L4` 对应未来限定白名单任务域中的高自动化运营模式
- `L5` 对应远期概念，不纳入当前版本的“已具备能力”描述

当前版本不应宣称：

- 当前系统已达到跨开放域任务的 L4 / L5
- 当前所有任务默认都可无人值守执行
- 当前所有节点族都具备统一成熟度的自动恢复与自动重规划能力
