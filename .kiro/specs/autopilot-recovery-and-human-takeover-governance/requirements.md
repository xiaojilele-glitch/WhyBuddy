# 需求文档：任务自动驾驶恢复机制与人工接管治理

## 目标

定义 WhyBuddy 在“任务自动驾驶”场景下的恢复机制与人工接管治理框架，使系统在偏航、失败、治理命中、质量不达标和外部依赖异常时，不会退化为不可解释的黑盒失败，而是能在自动恢复、降级执行、人工接手、恢复后继续和异常升级之间做出可审计、可回放、可治理的选择。

本 spec 的目标包括：

- 定义偏航检测、失败识别与恢复分类的统一口径
- 定义自动恢复、降级执行、人工接手与异常升级的边界
- 定义恢复后如何继续原任务、切换路线或进入重新规划
- 确保该机制与现有 `review / audit / revise / verify / HITL / runtime governance` 兼容
- 为后续 runtime、cockpit、takeover panel、replay、audit 提供一致语义

本 spec 不试图替代现有 Mission Runtime、HITL、decision、retry、resume、escalate、replan 基础设施，而是在其上建立一层统一的治理与恢复框架。

## 背景

当前仓库已经具备多类相关能力基础：

- 任务自动驾驶的高层对象定义
- `Drive State` 与 `replanning` 的高层状态机
- `Takeover Point`、接管面板与决策点抽象
- `MissionDecision`、`waiting / decision / resume / escalate` 人工链路
- `review / audit / revise / verify` 的质量与治理闭环
- 审计链、回放、证据链与治理面板
- runtime governance、成本治理、权限治理与异常控制
- 状态持久化、服务重启恢复与最小任务恢复链路

但在“任务自动驾驶真的跑起来”之后，仍存在一类尚未被统一建模的问题：

- 任务为什么算偏航，缺少统一判定口径
- 失败后先自动恢复还是直接请求人工，缺少一致策略
- 降级执行何时允许、何时禁止，缺少治理边界
- 人工接手后是替代系统继续，还是让系统恢复继续，缺少标准流程
- review / audit / verify 发现问题后如何回到执行链，缺少恢复语义
- runtime governance 命中后如何进入等待、升级或终止，缺少统一控制面

为了让任务自动驾驶具备“有方向盘、有刹车、有接管规则”的产品可信度，系统需要明确一套恢复与人工接管治理模型。

## 范围

本 spec 包含：

- 偏航、失败、阻塞、治理命中的定义
- 恢复阶段与恢复策略分类
- 自动恢复与降级执行策略
- 人工接手、人工接管升级与恢复后继续机制
- 与 review / audit / revise / verify / HITL / runtime governance 的兼容要求
- 恢复记录、治理记录、审计与回放要求

本 spec 不包含：

- 具体某个前端页面的完整视觉设计
- 对现有 runtime engine 的推翻式重写
- 第三方工单系统或外部审批系统接入
- 多人协同值班和排班治理
- 机器学习异常检测算法细节

## 核心术语

### 偏航（Deviation）

任务执行过程与当前 `Destination / Route / Governance Profile / Success Criteria` 出现显著偏离，继续按原路径推进的收益降低，且需要恢复、降级、改线或接管。

### 失败恢复（Failure Recovery）

针对执行失败、依赖失败、质量失败、状态异常或治理阻断的一组自动或半自动处理机制，用于让任务重新回到可推进状态，而不是直接失效。

### 降级执行（Degraded Execution）

在主路线、主能力、主资源不可用或不再合适时，系统在仍满足治理边界和最低交付要求的前提下，切换到更保守、更低成本、更低权限或更低自动化程度的执行方式。

### 人工接手（Human Takeover）

用户或人工角色对任务局部阶段或整体阶段进行显式接管，决定下一步策略、输入必要上下文、批准高风险动作或直接替代自动系统继续推进。

### 恢复后继续（Resume After Recovery）

任务经过自动恢复、人工决策、重新规划或降级执行后，回到可以继续推进的状态，恢复原路线、切换新路线或进入后续治理阶段。

### 治理边界（Governance Boundary）

系统在预算、权限、风险、质量、外部副作用、合规和自动化等级上的允许边界。恢复动作和降级动作都必须受其约束。

### 异常升级（Exception Escalation）

当自动恢复不可行、风险超过可授权范围、失败持续扩散、或人工接手仍无法决策时，将事件升级到更高权限的人类角色、治理流程或终止控制面。

## 需求

### 需求 1：系统必须定义统一的偏航与失败分类模型

系统必须对自动驾驶中的异常状态建立统一分类，避免不同模块各自发明“失败”“卡住”“需要接管”“需要重试”的口径。

系统至少应识别以下异常类型：

- 目标偏航：结果方向偏离 `Destination` 或用户最新意图
- 路线偏航：执行路径明显偏离当前 `Route`
- 质量偏航：中间结果或最终结果无法满足 `review / verify` 标准
- 治理偏航：预算、权限、风险或策略违反当前治理边界
- 依赖失败：工具、模型、执行器、外部服务或数据源不可用
- 状态阻塞：任务未完成，但已无法自动向前推进
- 恢复失败：既定恢复动作已耗尽但仍未回到可推进状态

验收标准：

- 文档必须明确每类异常的定义、触发信号和典型例子
- 文档必须区分“普通重试”“偏航”“阻塞”“需要重规划”
- 文档必须说明这些异常如何映射到 `executing / reviewing / blocked / takeover-required / replanning`

### 需求 2：系统必须支持统一的偏航检测信号来源

系统必须明确偏航检测不是单一模块判断，而是来自运行时、治理、质量和人工反馈的组合信号。

偏航检测至少应兼容以下信号来源：

- runtime 执行失败、超时、重试耗尽
- `review / verify` 发现结果不达标
- `audit` 或 governance 检测到权限、预算、风险边界命中
- Route 执行步骤与预期阶段不一致
- 用户手动指出方向错误、结果错误或策略不当
- 外部上下文变化导致当前路线不再适用

验收标准：

- 文档必须说明每类信号是“强触发”还是“弱触发”
- 文档必须说明多信号并发时的优先级处理原则
- 文档必须允许以事件投影方式重建偏航检测过程，用于 replay / audit

### 需求 3：系统必须定义恢复策略层级

系统必须按风险与代价，优先尝试从最小恢复动作到最重恢复动作逐层升级，而不是一出错就直接终止或直接人工接手。

恢复策略至少应包含以下层级：

- 局部重试
- 局部替代执行
- 路线内降级执行
- 阶段级重规划
- 人工确认后继续
- 人工接手执行
- 异常升级
- 终止任务

验收标准：

- 文档必须说明每一级策略的适用场景与退出条件
- 文档必须说明何时可以自动进入下一层，何时必须请求人工
- 文档必须说明恢复层级与 `retry / revise / replan / escalate / terminate` 的对应关系

### 需求 4：系统必须支持失败后的自动恢复

系统在低风险、低权限、低外部副作用的场景下，应优先尝试自动恢复，以减少不必要的人类打断。

自动恢复至少应支持：

- 节点级重试
- 替代工具、替代模型或替代执行器切换
- 读取缓存、快照或上一稳定状态恢复
- 跳过非关键步骤并继续
- 回退到上一个稳定阶段重新执行

验收标准：

- 自动恢复前必须检查当前治理边界是否允许
- 自动恢复必须记录触发原因、尝试动作、结果与耗时
- 自动恢复次数和预算必须受 runtime governance 约束
- 自动恢复失败后必须进入更高级别恢复、接管或升级流程，而不是无声卡死

### 需求 5：系统必须支持受治理约束的降级执行

系统必须把降级执行视为正式能力，而不是偷偷“凑合跑完”的兜底。

降级执行至少应覆盖：

- 降低模型等级或推理深度
- 降低并发度或缩小任务范围
- 关闭高成本增强能力
- 关闭高权限动作，改为只读或建议模式
- 从自动闭环切换为半自动协同模式

验收标准：

- 降级执行前必须明确降级原因、降级内容和预期影响
- 降级执行必须说明对时间、成本、质量、风险和自动化程度的影响
- 高风险、高副作用或高权限场景不得静默降级
- 降级执行必须进入 audit / replay，并能关联降级前后策略差异

### 需求 6：系统必须定义人工接手的触发边界

系统必须明确定义哪些场景必须由人工接手或人工确认，避免自动系统越权推进。

人工接手至少应在以下场景触发：

- 自动恢复预算耗尽
- 命中高风险、高权限或高成本治理边界
- `review / verify` 持续失败，且需要主观判断
- 任务结果可能影响真实业务决策或外部系统
- 目标、路线或风险接受需要显式责任归属
- 自动系统无法判定是继续、改线还是终止

验收标准：

- 文档必须区分“人工确认后继续”和“人工直接接手”
- 文档必须说明人工接手是局部阶段还是整任务范围
- 文档必须说明人工接手如何映射到现有 `Takeover Point` 和 `MissionDecision`

### 需求 7：系统必须支持人工接手后的恢复后继续

人工接手不应意味着自动系统彻底失效；系统必须支持人工介入后的继续执行。

人工接手后至少应支持以下继续方式：

- 按原路线继续
- 按人工修正后的路线继续
- 进入 `replanning` 后继续
- 以降级模式继续
- 进入 `reviewing` 或 `verify` 后再决定是否继续

验收标准：

- 人工决策结果必须能映射到 `resume()` 或 orchestrator 的继续链路
- 系统必须记录恢复前状态、人工动作、恢复后状态与关联理由
- 人工接手后若继续自动执行，系统必须重新校验治理边界

### 需求 8：系统必须与 review / audit / revise / verify 闭环兼容

恢复与接管机制必须兼容质量治理闭环，而不是只面向运行时报错。

至少应满足：

- `review` 发现问题时，可触发 revise、replan、degrade 或人工接手
- `verify` 失败时，可回到执行、修正或接管
- `audit` 命中治理问题时，可阻断继续、要求接管或升级
- `revise` 成功后，可重新进入 review、verify 或执行阶段

验收标准：

- 文档必须说明 `reviewing` 状态如何接入恢复策略
- 文档必须说明质量失败与运行失败在治理上的差异
- 文档必须说明恢复事件如何在 replay 时间线上被解释

### 需求 9：系统必须与 HITL / decision / wait-resume 链路兼容

本 spec 不应创建另一套与现有人工链路割裂的恢复系统。

至少应满足：

- 人工接手点必须能够映射到现有 `Takeover Point`
- 待人工处理时，系统必须能进入 `waiting` 或 `WAITING_INPUT`
- 人工处理提交必须复用或兼容 `submitMissionDecision()`
- 恢复后继续必须复用 `resume()`、orchestrator 决策后继续，或等价机制
- 需要人工升级时，必须能映射到 `escalate()`

验收标准：

- 文档必须给出与现有 `MissionDecision` 的兼容映射
- 文档必须说明恢复决策是否需要 `requiresComment`
- 文档必须说明异常接管如何复用现有接管队列

### 需求 10：系统必须与 runtime governance 兼容

恢复与接手机制必须受到 runtime governance 约束，而不是在异常状态下绕过治理。

至少应覆盖以下治理维度：

- 重试预算
- 预算上限与成本策略
- 权限边界
- 风险阈值
- 外部副作用限制
- 自动驾驶等级或自动化等级降级规则

验收标准：

- 每个恢复动作都必须说明是否受治理约束以及受哪些约束
- 治理命中时必须明确是阻断、建议、接管还是升级
- 治理决策必须进入审计记录

### 需求 11：系统必须定义异常升级边界

当自动恢复和普通人工接手都不足以安全解决问题时，系统必须支持异常升级。

异常升级至少应覆盖：

- 升级到更高权限的人工角色
- 升级到治理审批或人工值守流程
- 升级为强制停止并保留证据
- 升级为要求更高等级复核或审计确认

验收标准：

- 文档必须明确异常升级的触发条件
- 文档必须说明升级前必须保留哪些上下文与证据
- 文档必须说明升级后原任务状态如何冻结、继续或终止

### 需求 12：系统必须支持恢复与接管记录的可审计、可回放、可解释

恢复机制与人工接手必须成为任务证据链的一部分，而不是隐形的内部补救动作。

每次恢复或接手至少应记录：

- 触发时间
- 触发来源与异常分类
- 命中的治理边界
- 已尝试恢复动作
- 推荐动作与最终动作
- 人工输入、评论、批准或拒绝
- 恢复前后 Drive State
- 关联的 Route、Mission、Workflow、Runtime Event、Audit Event

验收标准：

- replay 中必须能看到任务何时偏航、何时恢复、何时接手
- audit 中必须能看到谁批准了什么恢复或降级动作
- 文档必须说明默认动作和自动恢复为什么被允许

### 需求 13：系统必须避免过度打断与过度自治

恢复治理既不能把所有问题都推给用户，也不能在高风险场景下静默自治。

验收标准：

- 低风险、低成本、低权限异常应优先自动恢复
- 中风险异常应优先建议接管或带默认建议的人机协同
- 高风险、高权限、高外部副作用异常必须显式人工确认
- 文档必须说明“不接手会发生什么”“允许自动恢复的前提是什么”

### 需求 14：系统必须支持分阶段落地

恢复与人工接管治理框架必须支持分阶段实现，而不是要求一次性重构全部 runtime。

建议阶段至少包括：

- 第一阶段：文档建模与语义统一
- 第二阶段：运行时信号标准化与恢复事件投影
- 第三阶段：接管面板、驾驶舱、replay、audit 接入
- 第四阶段：策略矩阵、治理自动化和灰度优化

验收标准：

- 文档必须说明哪些部分可先以 view model 或事件投影落地
- 文档不得要求当前主仓立即推翻现有 runtime / HITL / audit 实现

## 验收标准

- 存在一份明确的中文设计文档，说明偏航检测、恢复策略、人工接手与治理边界
- 文档中明确解释自动恢复、降级执行、人工接手、异常升级之间的区别
- 文档中明确说明与 `review / audit / revise / verify / HITL / runtime governance` 的兼容方式
- 文档中明确说明恢复后继续如何回到执行、重规划或复核链路
- 存在一份可渐进推进的任务清单，用于后续分阶段实现与审计收口

## 补充收口说明（2026-04-25，lane 6）

本轮继续按“设计直接收口 + 现有实现/测试有最小锚点”推进，本 spec 的要求层维持保守边界：

- 当前已经可以保守声称的部分
  - 偏航信号来源、触发强弱、恢复层级、人工确认/人工接手边界、治理结果分类、异常升级触发、与 `review / audit / revise / verify / HITL / runtime governance` 的兼容关系，已经能够在设计文档中被直接写清
  - 当前主仓也已经存在 recovery summary、takeover-required、waiting / replan 投影、mission decision 复用、runtime escalate / terminate、replay / audit mirroring 的最小事实链
- 当前仍不能外推为“已完成要求”的部分
  - recovery-specific option payload
  - 升级前证据保留统一合同
  - 恢复完成后回到 `executing / reviewing / delivered` 的独立 return-condition contract
  - recovery-specific audit record model
  - 恢复记录与 `decisionHistory / 任务历史 / audit` 的正式关联合同

因此，本 requirements 文档当前采取“可渐进推进”的收口口径：允许 design 与 tasks 在现有锚点上继续保守补勾，但不把尚未成型的 recovery 专用合同外推为已落地能力。

## 补充收口说明（2026-04-25，lane 6，二次推进）

本轮进一步把 requirements 的“当前主线最小闭环”边界写得更明确，避免后续把设计语义误读成必须先落地独立 recovery 子系统。

- 当前 requirements 对 `DeviationDetector`、`RecoveryCoordinator` 的要求
  - 应优先理解为“必须能被当前 shared summary + mission projection + wait-resume / escalate 主线重建的语义层”
  - 不强制要求先独立实现 detector 服务、coordinator 类、统一事件总线或统一恢复账本
- 当前 requirements 对展示与接入的要求
  - 优先要求服务端 projection、任务详情与 cockpit 详情变体能稳定读到 recovery / takeover / consequence / correlation
  - 不把独立 recovery cockpit、接手控制台、降级执行专门交互面板外推为本轮已完成范围
- 当前 requirements 对测试与灰度的要求
  - 允许先以 shared / runtime / decision / projection 的最小直接测试为锚点，收口单元测试设计与灰度原则
  - 不把动态阈值配置中心、统一 feature flag、恢复返回条件合同、recovery-specific audit model 外推为既有能力

因此，本 spec 当前最适合的验收口径是：

- 先确认恢复分类、治理兼容、状态映射、服务端投影、任务详情接入、测试策略与灰度边界已经写清楚；
- 再基于直接代码与直接测试，保守推进能被现实锚定的 tasks；
- 对 recovery-specific payload、证据保留合同、审计关联合同、持久化查询模型继续保持未完成状态。

## 补充收口说明（2026-04-25，lane 6，四次推进）

本轮继续收紧 requirements 与 design/tasks 的对应关系，重点把以下几类内容从“概念存在”推进到“已形成明确设计/测试计划合同”：

- 自动恢复
  - 节点级重试、替代执行器、快照/检查点恢复、跳过非关键步骤的准入条件与层级推进规则
- 降级执行
  - 降级影响表达、禁止静默降级场景、与 cost / permission / runtime governance 的对接方式
- 恢复返回条件
  - 何时回到 `executing / reviewing / delivered`，何时继续停留在 `blocked / takeover-required / replanning`
- 恢复记录
  - `RecoveryAttemptLedger` 的最小字段组与保留策略
- 测试与灰度计划
  - 集成验证矩阵、回归矩阵、连续性验证计划、阈值灰度分层与回滚触发条件

本轮要求层的保守口径如下：

- 可以视为“已收口”的部分
  - 文档已经把恢复治理规则写成可执行的设计定义或测试计划，不再只是抽象口号
  - 对应 tasks 若勾选，仅代表设计层、信息架构层、治理契约层或测试计划层已经成型
- 仍不能外推为“实现已完成”的部分
  - recovery-specific option payload
  - recovery-specific audit record model
  - 恢复记录持久化、查询接口、跨重启 attach
  - 独立 recovery cockpit、降级执行专用 UI、统一灰度平台

因此，本轮允许继续保守补勾以下类型的任务：

- 被 design 正文直接覆盖的设计任务
- 被测试矩阵、连续性验证计划、灰度/回滚规则直接覆盖的计划类任务

但仍不允许把以下类型任务提前勾选：

- 需要 recovery 专用写路径、专用 payload、专用 audit model 才能成立的实现任务
- 需要持久化层、查询接口、跨重启恢复语义直接证据才能成立的服务端任务
